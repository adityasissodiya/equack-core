\begin{abstract}
  Intermittently connected multi-stakeholder systems need authorization
  decisions that remain available offline yet converge deterministically
  after partitions heal.  \equack{} is an eventually consistent
  access-control model that records data operations, credential grants
  and revocations, and trust evidence in a single signed causal event

  \section{Introduction}
  \label{sec:introduction}
  Access control in multi-stakeholder environments faces an unresolved tension: devices and services must continue operating when disconnected from central infrastructure, yet security-critical decisions, especially credential revocation, must eventually be enforced consistently across all parties. The problem is sharpest in industrial Internet of Things (IoT)~\cite{MISTRY2020106382,liu2020fabriciot}, healthcare~\cite{HASSELGREN2020104040}, and cross-organizational data sharing~\cite{10.1145/3511899,DU2024101878}, where equipment changes hands between manufacturers, operators, and service providers throughout its lifecycle (e.g., a compressor sold by vendor~A, operated by mining company~B, and serviced by maintenance contractor~C, each holding distinct permissions over its control parameters).

  Federated research data sharing~\cite{HASSELGREN2020104040,10.1145/3511899},
  disaster-response coordination, and industrial control share an
  enforcement gap: independent authorities issue and revoke credentials,
  devices operate offline, and permissions evolve while nodes are
  disconnected.  Authorization in these settings is an
  \emph{offline-first} problem---nodes must decide locally, yet the
  system must later reconcile those decisions into a consistent,
  auditable state.

  Consider a factory running hundreds of programmable controllers on a
  restricted network.  Maintenance engineers from different vendors
  receive temporary access to specific machines, but controllers may
  operate for days without reliable
  connectivity~\cite{9551736,s23208535}.  An engineer updates a safety
  parameter while offline using a credential from their employer; hours
  later, that credential is revoked after a safety incident.  When logs
  from different controllers are synchronized, the system must
  deterministically agree on which parameter changes were
  authorized---and produce an audit trail linking each change to the
  actor, the permission, and the time.

  No existing approach provides deterministic post-partition
  reconciliation of fine-grained authorization decisions.
  Centralized systems require online coordination; token and
  capability schemes verify offline but cannot reconcile
  conflicting authorization state; Conflict-free Replicated Data Types (CRDTs) converge data but
  not authorization semantics; and ledger-based designs
  require consensus incompatible with disconnected operation
  (Section~\ref{subsec:gap-analysis}).

  Jacob and Hartenstein~\cite{jacob2024eventuallyac} formalize
  this setting as \emph{Eventually Consistent Access Control}
  (ECAC): replicas autonomously enforce access control from local
  knowledge and reconcile after reconnection. Their model distinguishes
  \emph{causal authorization} (immutable, based on predecessor history) from
  \emph{topological authorization} (mutable, based on the total execution order),
  and provides formal safety and liveness properties---including app state
  confluence under non-monotonic revocations---but does not specify
  how credentials are represented, how trust evidence propagates
  offline, how conflicting grants and revokes from independent
  issuers are resolved, or how fine-grained attribute-based
  policies are evaluated over reconciled history. They provide no
  implementation or experimental evaluation.
  Kleppmann et al.~\cite{kleppmann2019localfirst} independently
  identify access control---particularly revocation---as an
  unsolved problem for local-first CRDT-based software.

  This paper presents \equack{}, a concrete instantiation of the
  ECAC paradigm that closes these mechanism-level gaps. \equack{}
  records data operations, credential grants and revocations, and
  trust evidence in a single signed causal event log. Replicas
  merge logs by set union and reconstruct authorized state through
  deterministic replay with a deny-wins gate. Authorization is
  expressed through Verifiable Credential (VC)-based time-bounded epochs evaluated by a
  Cedar-inspired policy engine supporting Attribute-Based Access Control (ABAC) and Role-Based Access Control (RBAC) semantics,
  so that fine-grained, scoped, attribute-conditional permissions
  are enforceable offline and reconcilable after partition.

  The novelty is not the individual ingredients---CRDTs, VCs,
  policy engines---but the replay mechanism that composes them into
  a deterministic authorization function over reconciled history,
  providing properties (R2, R3) that no prior system achieves
  simultaneously with fine-grained policy expressiveness.

  This paper makes the following contributions.
\begin{enumerate}
    \item \textit{ECAC instantiation with fine-grained policy
      support} (Sections~\ref{sec:problem-analysis}--\ref{sec:ecac-model}).
      We instantiate the ECAC paradigm
      with VC-based authorization epochs, in-band trust propagation,
      and Cedar-inspired policy evaluation---mechanisms that ECAC
      defines abstractly but does not specify.

    \item \textit{Deterministic deny-wins reconciliation}
      (Section~\ref{sec:ecac-model}).  We define a causal Directed Acyclic Graph (DAG)
      linearization and replay gate that produces the same authorized
      state on every replica that has received the same events.
      Operations admitted under subsequently revoked credentials are
      excluded after reconciliation.

    \item \textit{Prototype and evaluation}
      (Sections~\ref{sec:ecac-model}--\ref{sec:evaluation}).  \textit{equack-core} is evaluated for convergence, revocation
      enforcement, multi-authority conflict resolution, audit integrity,
      and replay cost.  The experiments confirm identical state digests
      across 100 topological orderings of a 10\,000-operation log,
      suppression of all post-revocation writes, deterministic conflict
      resolution, tamper detection, and replay throughput up to
      45\,000~ops/s with 8--11$\times$ checkpoint speedup.
\end{enumerate}

  Section~\ref{sec:problem-analysis} formalizes the problem and derives requirements.
  Section~\ref{sec:ecac-model} presents the \equack{} model, its Rust prototype
  realization, and formal guarantees.
  Section~\ref{sec:evaluation} evaluates it.
  Section~\ref{sec:discussion} discusses trade-offs and positioning.
  Section~\ref{sec:conclusion} concludes.

\section{Problem Analysis}\label{sec:problem-analysis}

The offline multi-stakeholder authorization problem arises when systems must (i) continue operating during network partitions, (ii) support multiple independent authorities that can grant and revoke permissions, and (iii) eventually converge to a consistent, auditable authorization state. This section defines the problem context, presents conceptual requirements derived from motivating use cases, and evaluates existing approaches against those requirements.

The requirements and comparisons that follow assume an offline-first, multi-authority setting in which replicas may be disconnected for extended periods, multiple independent organizations issue and revoke credentials, and a computationally bounded adversary may delay, reorder, or partition messages. Section~\ref{subsec:system-model} gives the complete system and threat model; the brief characterization here is sufficient to interpret R1--R6 and the subsequent gap analysis. This setting explains why common fixes fail; online introspection endpoints violate offline-first operation; global consensus violates availability during partition; single-authority trust violates multi-stakeholder governance.

\subsection{The Offline Enforcement Problem}

Consider a set of replicas $R = \{r_1, \ldots, r_n\}$ operated by multiple stakeholders, maintaining shared resources under access control. Each replica may be offline for extended periods, during which it must make local authorization decisions. When replicas reconnect, they must reconcile their local states into a globally consistent outcome.

The core difficulty is not issuing permissions, standard Public Key Infrastructure (PKI) and token mechanisms suffice for that, but ensuring that all replicas eventually agree on \emph{which operations were authorized at the time they occurred}, despite:
\begin{itemize}
    \item \textit{Partition-time divergence}: Different replicas may observe different subsets of grants, revokes, and data operations during a partition.
    \item \textit{Out-of-order delivery}: A revocation may arrive at a replica after operations it should have invalidated.
    \item \textit{Multi-authority conflicts}: Independent issuers may make conflicting decisions (e.g., one grants while another revokes) that must be resolved deterministically.
\end{itemize}

\noindent Figure~\ref{fig:offline-enforcement-problem} illustrates these three challenges and the convergence pipeline that resolves them: after reconnection, replicas merge their event sets, establish a deterministic total order, and replay authorization decisions under a deny-wins policy to reach identical verdicts.

Jacob and Hartenstein~\cite{jacob2024eventuallyac} formalize this setting as ECAC (Section~\ref{sec:introduction}). Their abstract algorithm proves safety and liveness properties---including app state confluence under non-monotonic revocations---but operates at the level of Matrix's power-level-based authorization (LeABAC) rather than credential-based epochs with policy-engine evaluation. \equack{} instantiates their paradigm with concrete mechanisms.

\subsection{System and Threat Model}\label{subsec:system-model}

We consider a set of replicas operated by multiple stakeholders that jointly maintain a replicated log of operations over shared resources. Replicas are deployed in environments with intermittent connectivity (e.g., edge and industrial deployments) and may be offline for extended periods. Each replica locally stores all received operations and materializes application state by deterministic replay.

A \emph{subject} (principal) is an actor that initiates operations (users, services, devices). One or more independent \emph{issuers} create and revoke time-bounded credentials for subjects. Each replica hosts a policy enforcement point (PEP) that admits subject requests only when locally authorized, then emits a signed log event carrying the subject identity and authorization evidence. The event timestamp (HLC) and signature are produced by the replica/PEP, not the subject, preventing forging or backdating under our threat model. All state changes are represented as signed operations in a causal DAG, disseminated via gossip/anti-entropy over an asynchronous network where messages may be delayed, reordered, duplicated, or temporarily dropped.

\equack{} targets security failures arising from \emph{offline enforcement} and \emph{multi-authority governance}. We assume an adversary may (i)~compromise subject credentials and attempt unauthorized operations, (ii)~control the network to delay, reorder, or partition replicas, and (iii)~attempt to tamper with stored history. The adversary cannot forge replica/PEP signing keys or choose HLC timestamps, and is computationally bounded.

The following assumptions scope the guarantees:
\begin{enumerate}
    \item \textit{Offline-first operation.} Replicas may be disconnected for hours to days; solutions requiring continuous connectivity are excluded.
    \item \textit{Multi-authority governance.} Multiple independent organizations issue and revoke credentials; no single root authority controls all permissions.
    \item \textit{Honest-but-buggy administrators.} Issuers follow protocols but may make errors; \equack{} provides safety-by-default semantics.
    \item \textit{Eventual delivery.} Every well-formed operation is eventually delivered to every correct replica, though delivery order may be adversarially influenced.
    \item \textit{Bounded partition duration.} Offline periods are finite; replicas reconcile on reconnection.
    \item \textit{No physical key compromise.} Long-term private keys are not exfiltrated via physical attacks; compromise is modeled at the software/account level.
\end{enumerate}
\equack{} does not address fully Byzantine collusion, side-channel leakage, cryptographic primitive failures, or metadata privacy; these require additional mechanisms.

\subsection{Conceptual Requirements}\label{subsec:conceptual-requirements}

From the system context and motivating use cases (Section~\ref{sec:introduction}), we derive six conceptual requirements for offline multi-stakeholder access control. These serve as criteria for comparing existing paradigms and guiding \equack{}'s design.

\begin{enumerate}
    \item \textbf{R1---Offline enforceability.} Authorization decisions must be enforceable without contacting a central Policy Decision Point (PDP) or ledger; disconnected nodes operate using locally verifiable evidence.
    \item \textbf{R2---Deterministic post-partition convergence.} Given the same set of events, all parties must compute the same final authorization state regardless of message arrival order.
    \item \textbf{R3---Revocation safety under partition.} Revocations must deterministically invalidate effects in the replicated state after reconciliation, even if the revocation arrives after the operations it should have blocked. Revocation applies to the digital state; irreversible physical-world actions require compensating controls at the application layer.
    \item \textbf{R4---Tamper-evident accountability.} The history of grants, revokes, and authorization decisions must form an append-only, verifiable trace.
    \item \textbf{R5---Decentralized multi-authority issuance.} Independent issuers must grant and revoke permissions without coordination; verifiers check authenticity offline using locally available evidence.
    \item \textbf{R6---Confidentiality with post-revocation protection.} Unauthorized parties should not read protected content; after revocation-triggered key rotation, revoked principals cannot decrypt data created after their access ended.
\end{enumerate}

\subsection{Gap Analysis}\label{subsec:gap-analysis}

Table~\ref{tab:gap-summary} evaluates five dominant approaches against R1--R6. Centralized Identity and Access Management (IAM) couples enforcement to online coordination, failing R1 and R3 under partition~\cite{jacob2024eventuallyac}. Token-based schemes enable offline checks but lack deterministic reconciliation and timely revocation~\cite{rouhani2021distributed}. CRDTs converge data payloads; they do not define authorization semantics or revocation rollback~\cite{shapiro2011crdt,gomes2017_sec}. Blockchains offer total ordering and auditability at the cost of online consensus, which is incompatible with offline enforcement~\cite{8989788,9020182,9446108,8798730,9548956,DIFRANCESCOMAESA201993,s20010282,s22031271}. Capability-based systems~\cite{birgisson2014macaroons,ucan2022,andersen2019wave} verify delegated authority offline but cannot deterministically converge conflicting authorization state or retroactively revoke. No existing approach simultaneously satisfies R1--R6.

Table~\ref{tab:gap-summary} identifies which \emph{properties}
each paradigm satisfies. The deeper gap is mechanism-level: no
existing approach provides
(i)~deterministic reconciliation of authorization history after
    partition,
(ii)~deny-wins revocation semantics that retroactively exclude
    operations admitted under stale credentials,
(iii)~fine-grained policy evaluation (ABAC/RBAC) over the
    reconciled history,
(iv)~in-band propagation of trust and credential-status evidence,
    or
(v)~auditable linkage between credential state and data-state
    replay.
ECAC~\cite{jacob2024eventuallyac} provides the formal framework
for (i) but not the concrete mechanisms for (ii)--(v). \equack{}
fills these gaps.

\equack{} fills the gap between these approaches, CRDT replication provides data convergence but not authorization convergence~\cite{shapiro2011crdt,gomes2017_sec}; ledgers provide total order but require consensus; token/VC designs support offline checks but lack deterministic reconciliation~\cite{rouhani2021distributed}. \equack{}'s causal DAG linearization plus deny-wins gate makes authorization a reproducible function of the reconciled event set without online consensus.

Capability-based systems---Macaroons~\cite{birgisson2014macaroons}, UCAN~\cite{ucan2022}, Biscuit~\cite{biscuit2023}, WAVE~\cite{andersen2019wave}, and the Willow Protocol~\cite{willow2024}---enable offline verification of delegated authority (R1, R5) but do not address deterministic convergence of conflicting authorization state (R2) or retroactive invalidation of effects authorized under stale evidence (R3). Revocation relies on expiry or propagation, facing the same freshness problem as VC status lists. \equack{}'s contribution is orthogonal: it provides the replay mechanism that deterministically reconciles authorization state after partition healing.

Zanzibar~\cite{zanzibar2019} and its open-source derivatives (e.g., OpenFGA, SpiceDB) provide relationship-based authorization backed by strongly consistent stores, violating R1 by construction. Policy engines such as Open Policy Agent (OPA) and Cedar~\cite{cutler2024cedar} support offline policy \emph{evaluation} but not offline policy \emph{modification} with deterministic convergence. \equack{} sits between these extremes: offline operation with deterministic post-partition convergence, without global consensus.

Matrix's state resolution (v2)~\cite{matrix_stateresv2} is structurally similar---a DAG of events with deterministic state resolution across federated servers---but uses power-level-based authorization rather than CRDT formalism, deny-wins semantics, or VC-based epochs. Jacob and Hartenstein~\cite{jacob2024eventuallyac} formalize Matrix's approach as an ECAC instance; \equack{} builds on this foundation targeting industrial multi-stakeholder settings. \equack{} advances beyond the ECAC abstraction by specifying the credential format (W3C VCs), the conflict resolution semantics (deny-wins epochs), the policy evaluation engine (Cedar ABAC), and the trust propagation mechanism (in-band TRUST events).

The p2panda toolkit~\cite{p2panda2024} is the closest existing system. Its \texttt{p2panda-auth} crate~\cite{p2panda_auth2025} implements a convergent access-control CRDT with a ``strong removal'' resolver over a causal DAG, and its encryption layer~\cite{p2panda_encryption2025} uses Decentralized Continuous Group Key Agreement (DCGKA)-based group keys with forward secrecy via the Signal Double Ratchet. Three differences position \equack{} distinctly: (1)~\equack{} enforces deny-wins replay at the protocol level---the Gate function (Section~\ref{subsec:formal-model}) is mandatory and the materialized state always reflects revocations (Theorem~3), whereas p2panda treats retroactive data invalidation as an application-layer choice; (2)~\equack{} models authorization as VC-based epochs evaluated by a Cedar policy layer with multi-issuer trust propagated in-band, whereas p2panda uses group membership with four permission levels and no policy engine; (3)~\equack{} provides an explicit hash-chained audit trail (Theorem~4), whereas p2panda's tamper evidence is implicit in the signed DAG. p2panda's encryption layer is more mature than \equack{}'s experimental per-tag scheme.

\section{\equack{}: Model and Realization}
\label{sec:ecac-model}

Each subsection first describes the conceptual mechanism, then
shows how \textit{equack-core} realizes it. The formal model
(Section~\ref{subsec:formal-model}) is self-contained and does not depend on
implementation choices.

\equack{} is an Eventually Consistent Access Control model designed for offline-first operation with provable security and auditability guarantees.
It combines four elements: (1)~a causal event log with deterministic deny-wins replay, (2)~verifiable credentials for time-bounded authorization epochs, (3)~a hybrid Attribute-Based Access Control (ABAC)/Role-Based Access Control (RBAC) policy model with deny-overrides semantics, and (4)~per-tag encryption with key rotation.

The prototype, \textit{equack-core}, is a Rust library comprising approximately 8,500~lines (excluding tests). It uses Ed25519 for signatures, BLAKE3 for hashing, XChaCha20-Poly1305 for encryption, and SHA-256 for status list bitstrings. Operations are serialized in Concise Binary Object Representation (CBOR).

\subsection{Event Types and Causal DAG}

\equack{} maintains an append-only event log structured as a causal DAG, where each event points to its causal predecessors. The DAG encodes happens-before relations, and system state is derived by deterministically sorting and replaying events. A fixed traversal order ensures every replica produces the same sequence.

Concurrent events, those without causal order, are resolved using a composite key; a Hybrid Logical Clock (HLC) value plus a unique operation ID. The HLC combines physical and logical time to preserve causality; when timestamps match, op IDs break ties deterministically. Replay thus becomes a pure function of the event set, guaranteeing identical state across replicas (R2). 

\textbf{Separation of DAG and CRDT roles.} A common point of confusion in offline access control is the overlapping role of causal graphs and Data CRDTs. In \equack{}, the pipeline follows a strict separation of concerns, the DAG and topological sort fix a deterministic, totally-ordered sequence of events to evaluate \textit{authorization and policy validity}. Meanwhile, the CRDT state (e.g., MVReg, OR-Set) strictly handles \textit{data payload convergence}. The policy engine evaluates the strict ordered DAG to build epochs and TrustViews; only authorized DATA events are then handed off to the CRDTs. 


The event DAG acts as a convergent CRDT; merging logs is a set union, commutative, associative, idempotent. Operation structure and DAG mechanics are shown in Figure~\ref{fig:dag-structure}. Each operation carries a signed header with parent links, HLC timestamp, producer key, and typed payload. Replicas accept events opportunistically but only replay them once causal predecessors are present (activated/pending staging), ensuring every replica topologically sorts the same DAG prefix.

Replay follows a deny-wins rule, each event is applied only if the actor held the required permission at that moment; unauthorized events become no-ops. Operations are disseminated via gossip and anti-entropy (libp2p); this affects liveness but not safety or determinism.

\textbf{Realization.} The DAG is implemented in \texttt{core::dag}, which stores operations and computes the deterministic topological sort (\texttt{core::replay::topo\_sort}). CRDT state ($\mathrm{MVReg}$, $\mathrm{OR\text{-}Set}$) is managed by \texttt{core::crdt}. Operations are serialized as deterministic CBOR with Ed25519 signatures computed over BLAKE3 hashes of the unsigned header (\texttt{core::crypto}). Gossip dissemination and anti-entropy synchronization use libp2p (\texttt{net::gossip}, \texttt{net::sync}).

\subsection{Authorization Epochs and Credential Lifecycle}

\equack{} manages permissions using verifiable, time-bounded credentials. It adopts the W3C VC format for signed authorization tokens asserting a principal's rights over a specific interval (validFrom--validUntil). Credentials are tamper-evident and self-verifying; replicas independently confirm authenticity and validity~\cite{8603785}.

Revocations use standard VC status lists, cryptographically signed bitstrings indicating whether each credential ID is revoked. A valid credential defines an authorization epoch; expiry or revocation ends it.

Trust evidence is carried in-band as TRUST events (issuer keys and status-list updates) and replayed into a local TrustView, anchored by a pinned, cross-signed genesis checkpoint. The log forms a distributed PKI, an offline node can verify at any timestamp that a user held a valid, non-revoked credential without contacting a central authority.

During replay, the system checks for a valid credential covering each event's timestamp; if found, the operation applies, otherwise it is skipped.

Trust operations (IssuerKey, IssuerKeyRevoke, StatusListChunk) build the TrustView incrementally during replay. When a VC must be checked, replicas parse the JSON Web Token (JWT), look up the issuer key in TrustView, verify the Ed25519 signature, check status-list revocation, and extract claims, all from in-band log state, with no online PKI lookups.

Unlike the W3C Bitstring Status List specification, which assumes HTTP-accessible status endpoints, \equack{} propagates status-list updates as in-band $\mathsf{TRUST}$ events in the causal DAG. This sidesteps the need for always-online StatusList URLs, status freshness is bounded by event propagation latency rather than HTTP polling intervals. The trade-off is that status information is only as fresh as the latest $\mathsf{TRUST}$ event a replica has received. \equack{}'s design is compatible with W3C Bitstring Status List at the data-format level; deployments with partial connectivity can supplement in-band propagation with periodic HTTP fetches when available.

\textbf{Realization.} Epoch construction is implemented in \texttt{core::epoch}; VC parsing and Ed25519 signature verification in \texttt{core::vc}. TrustView assembly---materializing issuer keys and status-list evidence from replayed TRUST events---is implemented in \texttt{core::trust} and \texttt{core::trustview}. Status-list bitstrings are verified using SHA-256 checksums.

\subsection{Policy Evaluation and Deny-Wins Replay}

\equack{} uses a policy layer inspired by AWS's Cedar language (open-source ABAC). Policies may permit or forbid actions with a default deny stance; deny-overrides semantics ensure any matching forbid overrides all permits. The framework unifies RBAC and ABAC, roles are principal attributes, and Cedar's entity-attribute model supports hierarchical roles and contextual attributes. Policies are evaluated deterministically at each event's timestamp; identical inputs guarantee identical outcomes across replicas.

A REVOKE event deterministically terminates any authorization epoch opened by a prior or concurrent GRANT, so earlier permits cannot ``win'' due to message reordering. Revocation is not irreversible; a GRANT \emph{after} a REVOKE in the replay order opens a fresh epoch (re-grant). For concurrent GRANT/REVOKE pairs, the deterministic linearization chooses a single order.

\textbf{Replay Example.} Our compliance fixture contains an in-band \texttt{IssuerKey}, credential-and-grant pairs for two writers, 50 concurrent pre-revoke writes from each writer, a merge point, a \texttt{REVOKE} for one writer, and 10 post-revoke writes from each writer. Replay first builds the in-band TrustView, verifies both credentials, and opens epochs for both writers. The later \texttt{REVOKE} closes only writer~B's epoch in the deterministic replay order. As a result, all replicas apply both writers' pre-revoke writes, skip writer~B's post-revoke writes, and continue applying writer~A's writes.

Cedar's formal verification~\cite{cutler2024cedar} establishes decidability, totality, and soundness of authorization evaluation under a single consistent context where all entity data and policies are available and coherent at decision time. In \equack{}'s eventually consistent setting, this assumption holds locally (each replica evaluates deterministically given its local state) but not globally during partitions; two replicas may reach different permit/deny outcomes for the same request if one has received a forbid policy or revocation that the other has not. The deny-wins replay mechanism restores global consistency after convergence, once all replicas hold the same event set, replay re-evaluates every operation under the converged policy set, and the final materialized state $s^\star$ satisfies Cedar's deny-overrides semantics. Cedar's verified properties thus hold for $s^\star$ but not for the transient local decisions made during partitions. Formal verification of the replay mechanism itself, specifically that deterministic linearization plus deny-wins gating preserves the intended authorization invariants is future work and would require extending Cedar's Lean~4 proofs or independent mechanized checking.

The current model uses deny-overrides as the sole combining algorithm, any matching \texttt{forbid} policy overrides all \texttt{permit} policies. This is a deliberate safe default for multi-authority settings where no single authority should be able to override another authority's denial. Priority-based or temporal-precedence combining (e.g., last-writer-wins for policy updates) would require extending the epoch model to track policy precedence across authorities; we leave this for future work.

Entity attributes used in Cedar policy evaluation (e.g., role assignments, group memberships) may be stale during partitions, a replica that has not yet received an attribute-update event evaluates policies against outdated entity state. Because attribute changes are events in the causal DAG, they are replayed deterministically on convergence, and the final state $s^\star$ reflects the correct attribute values. Transient local decisions based on stale attributes are corrected during replay.

\textbf{Realization.} The deny-wins gate is implemented in \texttt{core::policy::gate}. Full replay (\texttt{core::replay::replay\_full}) and incremental replay from checkpoints (\texttt{core::replay::apply\_incremental}) both apply the gate as a pure fold over the linearized sequence. Policy evaluation uses a Cedar-inspired ABAC engine; Cedar's full type system is not implemented in the prototype.

\subsection{Tamper-Evident Audit Trail}

During replay, each replica emits a signed \emph{audit record} per DATA operation indicating applied/skipped status and reason. Records are linked in an append-only hash chain; any inconsistency is detectable relative to a pinned chain head digest (e.g., cross-signed checkpoint). Because replay is deterministic, independent replicas with the same event set compute identical audit traces and can validate each other after reconciliation (R4).

\textbf{Realization.} Audit chain management is implemented in \texttt{store::audit}. The \texttt{replay\_full\_with\_audit} entry point accepts an \texttt{AuditHook} trait object, emitting signed records during replay. Checkpoints (state snapshot plus topological watermark) are managed by \texttt{store::checkpoint} backed by RocksDB.

\subsection{Confidentiality and Key Rotation}

\equack{} supports an optional per-tag payload encryption layer. Each DATA operation carries ciphertext $c := \mathsf{Enc}_{K_{\tau}^{(v)}}(m)$ with key version $v$ (XChaCha20-Poly1305). Ciphertext is replicated for convergence; access control is enforced at \emph{query time} via the deny-wins gate, with unauthorized subjects receiving a redacted view. On revocation, a KeyRotate event advances to $K_{\tau}^{(v+1)}$, distributed in-band via KeyGrant records encrypted to authorized subjects' public keys. This provides \emph{leaver security}; revoked subjects lacking $K_{\tau}^{(v+1)}$ cannot decrypt post-rotation ciphertexts. This component is experimental; we treat confidentiality results as indicative.

\textbf{Realization.} Encryption and decryption use XChaCha20-Poly1305 in \texttt{core::crypto}. Per-tag key versioning and KeyGrant/KeyRotate handling use X25519 key agreement without a full ceremony protocol. Tag keys are persisted by \texttt{store::put\_tag\_key} / \texttt{store::get\_tag\_key}.

\subsection{Developer Interface}\label{subsec:developer-interface}

The following pseudocode summarizes the core operations a developer calls against \textit{equack-core}. The signatures are simplified from the actual Rust API.

Engineer~E submits parameter updates via \texttt{op\_data} while offline. When vendor~A revokes E's credential after a safety incident, the \texttt{op\_revoke} event propagates via gossip synchronization. \subsection{Formal Model and Guarantees}\label{subsec:formal-model}

Let $\mathsf{Subj}$ be subjects (principals), $\mathsf{Res}$ resources, $\mathsf{Act}$ actions, and $\mathsf{Attr}$ attribute valuations.
Physical time is measured in milliseconds as $(T_{\mathrm{ms}}=\mathbb{N}, \le)$.
Let $\mathsf{NodeId}$ be replica identifiers, and let hybrid logical clock timestamps range over
$T_{\mathrm{hlc}} := T_{\mathrm{ms}} \times \mathbb{N} \times \mathsf{NodeId}$ ordered lexicographically.
Let $\mathrm{phys} : T_{\mathrm{hlc}} \to T_{\mathrm{ms}}$ project the physical millisecond component.
We use $T_{\mathrm{ms}}$ for credential validity intervals and status freshness, and $T_{\mathrm{hlc}}$ (via lexicographic order) only for deterministic linearization.
Let $\mathsf{H}$ be a collision-resistant hash and $\mathsf{Sig}_k(\cdot)$ a secure signature scheme.

\medskip
\noindent\textbf{Verifiable credentials and trust.}
A verifiable credential (VC) is a tuple
\[
  c = \langle s,\, \alpha,\, V,\, \mathrm{nb},\, \mathrm{na},\, \sigma_I\rangle
\]
where $s\!\in\!\mathsf{Subj}$, $\alpha\!\in\!\mathsf{Attr}$ (roles/scopes), $V$ issuer identifier,
and $\mathrm{nb},\mathrm{na}\!\in\!\mathbb{T}$ are not-before/after bounds.
$\sigma_I$ is a signature under issuer key $K_V$.
A status list is a signed, time-versioned snapshot published by issuer $V$ that maps credential identifiers to $\{\mathsf{good},\mathsf{revoked}\}$.
We model status evidence as a sequence of $\mathsf{TRUST}$ events that carry status-list updates (or chunks).
Let $\mathsf{SL}^{i}_{V}$ denote the status map materialized by replaying all $\mathsf{TRUST}$ events with $\mathrm{pos}(e) \le i$ and taking the latest published status for each credential id.
When validating a credential for an event $e$, we query $\mathsf{SL}^{\mathrm{pos}(e)}_{\mathrm{iss}(c)}(\mathrm{id}(c))$.
A \emph{TrustView} event carries (i) issuer keys $\{K_V\}$ and (ii) status-list updates, recorded in-band so replicas can validate VCs offline using the latest status snapshot up to $t$ in the replay order.

\medskip

Events are partitioned as
\[
  \mathsf{Ev} \;=\; \mathsf{DATA}\;\uplus\;\mathsf{GRANT}\;\uplus\;\mathsf{REVOKE}\;\uplus\;\mathsf{TRUST}\;\uplus\;\mathsf{KEY}.
\]
The subset $\mathsf{KEY}$ contains key-management events (e.g., \textsc{KeyRotate}, \textsc{KeyGrant}) that update per-tag key state used for query-time decryption.
Each event $e\in\mathsf{Ev}$ has an \emph{unsigned} header $h_0(e)$ containing domain fields, causal parents $\mathsf{par}(e)\subseteq\mathsf{Ev}$, a hybrid logical clock $\mathrm{hlc}(e)\in\mathbb{T}$ assigned by the producing replica, the producer public key $\mathrm{pk}_r$, and the payload (or a payload hash). We define the event identifier as
\[
  \mathsf{op\_id}(e) := \mathsf{H}(\mathrm{domain}\,\|\,\mathrm{CBOR}(h_0(e))).
\]
The producing replica signs this identifier:
\[
  \sigma_e := \mathrm{Sig}_{\mathrm{sk}_r}(\mathsf{op\_id}(e)).
\]
Signing $\mathsf{op\_id}(e)$ is sufficient because $\mathsf{op\_id}(e)$ commits to the canonical bytes of $h_0(e)$ under collision resistance of $\mathsf{H}$.
The full header is $h(e) := (h_0(e), \sigma_e)$. This avoids any circularity; $\mathsf{op\_id}(e)$ is derived from the unsigned header, and $\sigma_e$ covers $\mathsf{op\_id}(e)$.
The \emph{signed, hash-linked op log} is an append-only bag of events with edges $e'\!\to\! e$ for $e'\in\mathsf{par}(e)$.
The induced graph $G=(\mathsf{Ev},E)$ must be a DAG; this causal DAG defines the partial order $\prec$ as the transitive closure of $E$.

\medskip

Let the deterministic key be
\[
  k(e) := \big(\mathrm{hlc}(e),\mathsf{op\_id}(e)\big),
\]
ordered lexicographically. Define the replay sequence $\langle e\rangle_{\triangleleft}$ as the output of Algorithm~\ref{alg:linearize} applied to the causal DAG $G$. We then define $e_i \triangleleft e_j$ iff $e_i$ appears before $e_j$ in $\langle e\rangle_{\triangleleft}$.

\begin{algorithm}[!htbp]
\caption{Deterministic linearization of a causal DAG}
\label{alg:linearize}
\begin{algorithmic}[1]
\Require DAG $G=(V,E)$, key $k(e)=(\mathrm{hlc}(e),\mathsf{op\_id}(e))$
\Ensure Sequence $L$ that is a deterministic linear extension of $\prec$
\State $\mathrm{indeg}[v] \gets |\{(u,v)\in E\}|$ for all $v\in V$
\State $Q \gets$ min-priority queue ordered by $k(\cdot)$
\ForAll{$v\in V$ with $\mathrm{indeg}[v]=0$}
  \State $Q.\mathrm{push}(v)$
\EndFor
\State $L \gets [\ ]$
\While{$Q$ not empty}
  \State $v \gets Q.\mathrm{pop\_min}()$
  \State append $v$ to $L$
  \ForAll{$(v,w)\in E$}
    \State $\mathrm{indeg}[w] \gets \mathrm{indeg}[w]-1$
    \If{$\mathrm{indeg}[w]=0$}
      \State $Q.\mathrm{push}(w)$
    \EndIf
  \EndFor
\EndWhile
\State \Return $L$
\end{algorithmic}
\end{algorithm}

\noindent Let $\langle e \rangle_{\triangleleft} = [e_0,\dots,e_{n-1}]$ be the output of Algorithm~\ref{alg:linearize}.
Define the deterministic replay position $\mathrm{pos}(e_i) := i$.

If two events $e_A$ and $e_B$ are concurrent and both are ready (zero indegree) at some step, Algorithm~\ref{alg:linearize} selects the smaller key, if $\mathrm{hlc}(e_A)<\mathrm{hlc}(e_B)$ then $e_A$ is chosen first; if equal, the smaller $\mathsf{op\_id}$ is chosen.


\medskip

Epochs are computed deterministically by replaying non-\textsc{DATA} events in $\langle e \rangle_{\triangleleft}$.
A \textsc{GRANT} event $g$ at position $p_g := \mathrm{pos}(g)$ that references a valid credential opens an epoch record
$\mathsf{ep} = \langle s,a,\mathrm{scope}, p_{\mathrm{start}}{=}p_g, p_{\mathrm{end}}{=}\infty, \mathrm{nb}, \mathrm{na} \rangle$.
A \textsc{REVOKE} event $r$ at position $p_r := \mathrm{pos}(r)$ closes matching open epochs by setting
$p_{\mathrm{end}} := \min(p_{\mathrm{end}}, p_r)$.
We write $\mathsf{ep}\in \mathsf{Epochs}(s,a)$ for epochs associated with subject $s$ and action $a$.

\medskip
For subject $s$, scope $a\!\in\!\mathsf{Act}$, and event $e$, define the authorization predicate
\[
\begin{aligned}
\mathsf{Auth}(s,a,e) := \exists c\ \text{s.t.}\ & \mathrm{phys}(\mathrm{hlc}(e)) \in [\mathrm{nb},\mathrm{na})\ \land\ c \text{ grants }(s,a)\\
& \land\ \mathsf{SL}^{\mathrm{pos}(e)}_{\mathrm{iss}(c)}(\mathrm{id}(c))=\mathsf{good}.
\end{aligned}
\]
Status-list Time-to-Live (TTL) is a freshness hint for operational policy; our determinism guarantees assume replicas replay the same TrustView history, while deployments may additionally deny decisions when status evidence is older than a configured threshold.

\medskip

Let $\mathcal{P}$ denote the Cedar policy, modeled as a total function
\[
  \mathcal{P} : \mathsf{Subj}\times\mathsf{Act}\times\mathsf{Res}\times\mathsf{Attr}\to\{\mathsf{allow},\mathsf{deny}\},
\]
with \emph{deny-overrides} semantics.
For a $\mathsf{DATA}$ event $e$ with tuple $(s,a,r,\alpha_e)$, define:
\[
\begin{aligned}
\mathsf{Gate}(e) :=\;&
\Big(
    \exists\, \mathsf{ep} \in \mathsf{Epochs}(s,a)
    \;\text{s.t.}\; \\
&\qquad
    \mathrm{pos}(e) \in [p_{\mathrm{start}}, p_{\mathrm{end}})
    \;\wedge \\
&\qquad
    \mathrm{phys}(\mathrm{hlc}(e))
    \in [\mathrm{nb}, \mathrm{na})
\Big) \\
&\wedge\;
\mathcal{P}(s,a,r,\alpha_e) = \mathsf{allow}.
\end{aligned}
\]
If \emph{any} deny applies, $\mathcal{P}$ returns $\mathsf{deny}$ and the DATA op is skipped (deny-wins).

\medskip

Let $(\mathcal{S},\bullet)$ be the state space with a deterministic update function
$U:\mathcal{S}\times\mathsf{DATA}\to\mathcal{S}$.
We assume the DATA payloads target CRDTs such as $\mathrm{MVReg}$ and $\mathrm{OR\!-\!Set}$ whose effect under the same total order is deterministic.
Given initial state $s_0$, define \emph{deterministic replay} over the sequence $\langle e\rangle_{\triangleleft}$:
\[
\begin{aligned}
\mathsf{Replay}(s_0, \langle e\rangle_{\triangleleft}) :=\ &
\mathrm{fold}\Big(s_0,\ \lambda(s,e).\\
& \begin{cases}
  U(s,e) & \text{if } e\in\mathsf{DATA}\ \land\ \mathsf{Gate}(e)\\
  s      & \text{otherwise}
  \end{cases}\Big).
\end{aligned}
\]
The \emph{materialized state} is $s^\star := \mathsf{Replay}(s_0,\langle e\rangle_{\triangleleft})$.
Checkpoints store $(\mathsf{H}(s^\star),\mathrm{meta})$ periodically together with the append-only event store.

\medskip
Define the audit stream as a sequence of records
\[
\begin{aligned}
\mathsf{Aud}(i) &= \big\langle e_i,\ \mathsf{Gate}(e_i),\ \mathsf{H}(s_i)\big\rangle,\\
\mathrm{ch}_0 &:= \bot,\quad
\mathrm{ch}_{i+1} := \mathsf{H}\!\big(\mathrm{ch}_i \,\|\, \mathsf{Aud}(i)\big),
\end{aligned}
\]
signed by the auditing replica (producer) to yield a tamper-evident chain $(\mathrm{ch}_i)$ of applied/skipped decisions.

\medskip
Let $\mathsf{Tag}$ denote data tags; each tag $\tau$ has a versioned symmetric key $K_{\tau}^{(v)}$. A DATA payload $m$ is stored as $(c,v)$ where $c := \mathsf{Enc}_{K_{\tau}^{(v)}}(m)$. Ciphertexts are replayed and replicated for convergence; authorization controls \emph{decryption} at query time. On a KeyRotate event for $\tau$ at time $t_r$, the current key version advances to $v{+}1$ and subsequent writes use $v{+}1$; KeyGrant records distribute $K_{\tau}^{(v+1)}$ only to subjects authorized to read $\tau$ after $t_r$.

Events are disseminated via secure anti-entropy gossip; missing parents are fetched until $G$ is complete.
This affects \emph{when} a replica can replay but not \emph{what} $s^\star$ is, once the same set of events is present.

\paragraph*{\textit{Pre-convergence safety window}} Let $r$ be a revocation event created at physical time $t_r$ on some replica. Define the \emph{safety window} $W(r)$ as the interval $[t_r, t_{\text{heal}}]$ where $t_{\text{heal}}$ is the earliest time at which all correct replicas have received $r$ via anti-entropy. During $W(r)$, replicas that have not yet received $r$ may accept operations from the revoked principal under stale authorization evidence. Under the bounded partition assumption (Section~\ref{subsec:system-model}, item~5), $W(r) \leq \Delta_{\text{partition}} + \Delta_{\text{propagation}}$ is finite.

\equack{} does not prevent tentative authorization during $W(r)$; it guarantees that the materialized state $s^\star$ computed after reconciliation reflects $r$. Specifically, any $\mathsf{DATA}$ event $e$ from the revoked subject with $\mathrm{pos}(e) \geq \mathrm{pos}(r)$ in the deterministic order $\triangleleft$ is skipped by the deny-wins gate, regardless of which replica originally accepted $e$. This provides \emph{retroactive revocation safety}, the final digital state is consistent with the revocation, even though individual replicas may have temporarily acted on stale evidence. In contrast, Zanzibar's Zookie mechanism~\cite{zanzibar2019} provides real-time revocation at the cost of availability during partition (consistent with Consistency--Availability--Partition tolerance (CAP) theorem constraints). \equack{} trades real-time revocation for availability, repairing safety on merge.

\subsection{Main Theorems}\label{subsec:main-theorems}

R1--R6 in Section~\ref{subsec:conceptual-requirements} are \emph{system-level requirements} (deployability and security objectives) for offline-first, multi-stakeholder settings.
The theorems below are \emph{formal guarantees} about \equack{}'s deterministic replay semantics and audit/confidentiality mechanisms under the model assumptions.
The gap analysis (Table~\ref{tab:gap-summary}) maps R1--R6 to paradigm coverage; the theorems below formalize the guarantees, and the evaluation (Section~\ref{sec:evaluation}) provides empirical evidence.

\noindent\textbf{Theorem~1 (Determinism).}
\textit{For any DAG $G$ and initial state $s_0$, the order $\triangleleft$ is unique and
$\mathsf{Replay}(s_0,\langle e\rangle_{\triangleleft})$ yields a unique $s^\star$.}


Algorithm~\ref{alg:linearize} is Kahn's topological sort with a deterministic min-heap over the ready set. For a fixed DAG $G$ and a total key order $k$, it produces a unique linear extension $\langle e\rangle_{\triangleleft}$. Replay is a pure fold over this sequence with a deterministic update function, hence yields a unique $s^\star$.

\medskip
\noindent\textbf{Theorem~2 (Convergence).}
\textit{Fix bootstrap anchors $B$ consisting of a pinned, cross-signed genesis checkpoint digest (and the stakeholder public keys used to validate it).
Let two replicas start from the same anchors $B$ and ingest the same multiset of well-formed events (including $\mathsf{TRUST}$ and $\mathsf{KEY}$ events).
Then their materialized states are byte-identical: $s^\star_1 = s^\star_2$.}


Both replicas begin from the same anchored initial trust/policy state implied by $B$, ingest the same event set, and thus compute the same causal DAG $G$.
Algorithm~\ref{alg:linearize} yields the same $\triangleleft$ order, hence the same $\mathrm{pos}(\cdot)$ assignments, the same epoch construction, and the same deny-wins gate decisions.
Applying the same ordered sequence of authorized \textsc{DATA} updates under deterministic $U$ yields identical $s^\star$.

\medskip
\noindent\textbf{Theorem~3 (Policy-Safety, deny-wins).}
\textit{Every effect in $s^\star$ originates from a $\mathsf{DATA}$ event $e$ such that $\mathsf{Gate}(e)$ held at position $\mathrm{pos}(e)$.
If any deny applies (policy or missing epoch), $e$ has no effect.}


By definition of $\mathsf{Replay}$: unauthorized DATA are skipped; deny overrides return $\mathsf{deny}$, hence no state transition occurs.

\medskip
\noindent\textbf{Theorem~4 (Audit tamper-evidence).}
\textit{Let $\mathrm{ch}_N$ be a pinned audit-chain head digest for the first $N$ records (obtained out-of-band or via cross-signing). If the audit chain verifies and the computed head equals $\mathrm{ch}_N$, then any modification to the first $N$ audit records (deletion, insertion, reordering, or bit-flip) is detectable with overwhelming probability.}


The head digest is a collision-resistant hash of the prefix. Any change to the prefix changes the computed head unless the adversary finds a collision/second preimage or forges a signature.

\medskip
\noindent\textbf{Theorem~5 (Leaver security).}
\textit{Fix a tag $\tau$ and a rotation event at time $t_r$ that advances to key version $v{+}1$. Assume indistinguishability under chosen-plaintext attack (IND-CPA) encryption and that $K_{\tau}^{(v+1)}$ is delivered only to subjects authorized for $\mathrm{read}_\tau$ at times $\ge t_r$. Then any subject not authorized for $\mathrm{read}_\tau$ at times $\ge t_r$ cannot decrypt ciphertexts produced under version $v{+}1$, except with negligible probability.}


By assumption, unauthorized subjects do not obtain $K_{\tau}^{(v+1)}$. Under IND-CPA security, ciphertexts under an unknown key reveal no information about plaintext beyond negligible probability.

\section{Evaluation}
\label{sec:evaluation}

We evaluate \equack{} through deterministic experiments that provide
empirical evidence for the main theorems (Theorems~1--5,
Section~\ref{subsec:main-theorems}) and quantify replay cost.
Table~\ref{tab:experiment-summary} maps each experiment to the
theorem(s) it targets. All experiments use fixed seeds and a
deterministic environment (\texttt{TZ=UTC, SOURCE\_DATE\_EPOCH=1}).
Offline operation without continuous global consensus is a
precondition of the model, not an experimental variable.

\subsection{Methodology}\label{subsec:methodology}

Experiments ran on a single Linux x86\_64 workstation using Rust 1.85 in release mode. Multi-node convergence (E12) uses in-process deterministic simulation; partition behavior is validated via offline log merges. State digests are computed as $\mathrm{blake3}(\mathrm{DOMAIN}\,\|\,\mathrm{canon}(\textit{state}))$, where $\mathrm{canon}(\cdot)$ is a deterministic byte serialization with stable key ordering.

We use three synthetic workloads (Table~\ref{tab:workloads}) and two policy-focused CBOR logs. The \emph{hb-chain} workload is a single writer appending sequentially to one key, giving a baseline with no concurrency or policy overhead. The \emph{concurrent} workload has eight writers modifying the same key, stressing the MVReg CRDT's conflict resolution. The \emph{offline-revocation} workload mixes writes with mid-workload revocations, forcing epoch recalculation during replay. Two additional logs exercise deny-wins behavior: \texttt{e3-policy.cbor} (grant, 500 writes, revoke, 500 writes) and \texttt{e4-policy.cbor} (multi-issuer grants, revoke, post-revoke writes). For the partition scenario, logs \texttt{partition-A.cbor} and \texttt{partition-B.cbor} were generated and merged to model revocation propagation on reconnection.

\subsection{Correctness}\label{subsec:correctness}

\paragraph*{\textit{Convergence (E1; T1, T2)}}
A 10{,}000-op concurrent-workload log was permuted into 100 valid orderings (respecting causal dependencies) and each was replayed independently. All 100 produced the same digest (\texttt{\small cbab89ee\ldots b7f493b2}), with mean replay time 615~ms ($\sigma=18$~ms). Delivery order does not affect the final authorized state---a prerequisite for safe offline operation across independently administered replicas.

\paragraph*{\textit{Deny-wins revocation (E3; T3)}}
User $U$ was granted write access, performed 1{,}000 writes, and had the credential revoked after write~500. The 1{,}004-op log (1{,}000 Data + IssuerKey, Credential, Grant, Revoke) was replayed: the first 500 writes applied, the remaining 500 skipped. Digest: \texttt{\small 8da8b49e\ldots 46633fbf}. An engineer whose credential is revoked after a safety incident has all post-revocation parameter changes excluded from the converged state, regardless of which controllers received the revocation first.

\paragraph*{\textit{Revocation under partition (E11; T2, T3)}}
In partition~A a user is revoked; partition~B, which has not yet received the revocation, contains further writes from the same user under stale evidence. After merging both logs and replaying, the 50 pre-revocation data writes (from both partitions) were applied and all 10 post-revocation writes from partition~B were skipped. A revocation issued on one side of a partition retroactively invalidates stale writes accepted on the other side.

\paragraph*{\textit{Multi-authority conflict (E4; T1, T3)}}
Three issuers ($I_1$, $I_2$, $I_3$) with overlapping authority grant user $U$ the ``editor'' role at $t_1$, $t_2$; $I_3$ revokes at $t_3>t_2$. The resulting 128-op log was injected into 10 replicas in random orders. All converged to the same digest (\texttt{\small f2fbb024\ldots 1103c2b2}): writes during $[t_1,t_3)$ accepted, writes after $t_3$ skipped.

\paragraph*{\textit{Audit integrity (E5; T4)}}
An audited log was generated and the chain head digest $\mathrm{ch}_N$ was recorded. Three tamper modes were applied: (1)~a byte flip in one record's payload, (2)~deletion of one record from the pinned prefix, and (3)~a swap of two adjacent records. Chain verification against $\mathrm{ch}_N$ succeeded on the untampered log and failed on all three mutations, detecting each at the exact tampered record. Any party can independently verify that no authorization decisions were altered after the fact.

\paragraph*{\textit{Cross-platform determinism (E2; T1)}}
A 127-op CBOR fixture exercises all event categories (Data, Grant, Revoke, Trust) and includes true concurrency. A Continuous Integration (CI) test replays it and asserts the final digest equals a hardcoded constant (\texttt{\small 4ea9684a\ldots bdc2b9}, blake3). Any replay-engine change that alters the digest breaks the build, following standard cross-environment consistency verification practice. Full cross-platform validation (aarch64, macOS, Windows) awaits CI infrastructure.

\paragraph*{\textit{Multi-node sync (E12; T2, T3)}}
In an in-process simulation, three nodes each generated 1{,}000 ops independently, then synced; all converged. A second scenario partitioned node~C while A issued a Revoke targeting C. After healing and full sync, all three nodes held the same digest, with C's post-revoke ops gated out (Table~\ref{tab:e12-sync}). This demonstrates end-to-end deny-wins enforcement across independently operating nodes.

\paragraph*{\textit{Confidentiality overhead (E13, E14; T5)}}
Per-tag encryption adds $\sim$6.5\% replay overhead (76.8 vs.\ 81.8~ms on 100 writes) with 40~bytes/payload ciphertext expansion. Key rotation costs $\sim$78~ms per KeyRotate event. These are small-scale measurements; larger logs are needed to separate per-op cost from startup overhead.


\subsection{Limitations and Threats to Validity}\label{subsec:limitations}

Disconnected nodes cannot learn of revocations until they sync~\cite{10.1007/978-3-030-01950-1_48}. The vulnerability window equals partition duration: operations written by a revoked user are invalidated on reconnection, but not before. The append-only log grows monotonically, and the full log must be retained for audit. Log compaction with verifiable snapshots could bound storage but remains future work. Long-offline nodes face a proportionally large replay on reconnection; checkpoints help but do not eliminate this. Per-tag encryption protects payloads, yet operation structure, timestamps, and producer keys remain visible; full metadata privacy would require anonymous credentials, Attribute-Based Encryption (ABE), or zero-knowledge proofs~\cite{8888155,zhang2020attribute}. Revocation also cannot undo physical-world actions already taken (e.g., an actuator that has fired). Applications needing staged commit must handle this themselves.

Regarding validity: E6 covers six log sizes with five trials each, but E8 and E9 use fewer sizes and a single workload. All numbers come from one Linux x86\_64 machine, and E12 uses in-process simulation rather than real network transport. The workloads are synthetic; validation on production traces is future work. E2 determinism is verified on one platform; cross-platform validation (aarch64, macOS, Windows) awaits CI infrastructure.

\section{Discussion}
\label{sec:discussion}

\paragraph*{\textit{Partition-duration trade-off and safety window}}
\equack{}'s deny-wins gate repairs authorization state after partition heal but cannot prevent unauthorized effects during the partition itself. The vulnerability window equals partition duration ($W(r) \leq \Delta_{\text{partition}} + \Delta_{\text{propagation}}$): a revoked user's writes are retroactively invalidated when logs merge, yet physical-world consequences (actuator commands, parameter changes) persist. For short disconnections (minutes to hours), operators can defer safety-critical actuations until sync confirms authorization; for day-long partitions, application-layer compensating controls (staged commit, human-in-the-loop) are needed. The guarantee is that the final digital state $s^\star$ is always consistent with the revocation---the physical world cannot be rewound.

\paragraph*{\textit{Scalability and deny-wins conservatism}}
Memory grows at $\approx$1.7~KB/op (Table~\ref{tab:e9-memory}), so a 256\,MB edge device holds roughly 150K operations---enough for weeks of maintenance logs, but not high-frequency telemetry. Throughput drops from 45K~ops/s on linear chains to 7K~ops/s with eight concurrent writers (Table~\ref{tab:e7-throughput}); at $\approx$430~bytes/op in CBOR, a million-operation log occupies $\sim$430\,MB before write amplification. Without log compaction (deferred to future work), long-lived deployments will need archival. These constraints position \equack{} for authorization-rate workloads on devices with modest storage rather than high-throughput data-plane enforcement.
The deny-wins semantics are deliberately conservative: concurrent GRANT and REVOKE resolves in favor of revocation, appropriate where over-granting is riskier than over-denying. An erroneous revocation retroactively voids subsequent work; recovery requires a new GRANT to open a fresh epoch (Section~\ref{subsec:formal-model}), which may not be timely during disconnection. Making the combining algorithm configurable per scope is a natural extension, though it would complicate the convergence argument.

\paragraph*{\textit{In-band trust and comparison scope}}
Carrying issuer keys and status-list updates as replicated log events eliminates online PKI lookups, enabling offline VC verification. The cost is that TrustView grows with each key registration and status-list chunk; deployments with frequent key rotations across many issuers would see meaningful log growth from trust material alone. The pinned genesis checkpoint anchors the trust chain; its governance (who signs, how keys are distributed, how organizations join or leave) is an open design problem.
Tables~\ref{tab:gap-summary} and~\ref{tab:mech-compare} evaluate \equack{} against paradigm categories rather than specific systems, because no existing system combines CRDT replication with in-band VCs and a deny-wins gate. The key distinction from application-level access control is enforcement timing: application-level checks execute at ingest and cannot retroactively invalidate operations accepted under stale evidence; \equack{}'s replay-time gate can (R3). The closest system is p2panda, whose retroactive replay of application data is optional and application-implemented, whereas \equack{}'s deny-wins gate is mandatory during deterministic replay. A direct empirical comparison is infeasible as p2panda provides no peer-reviewed formal model or published benchmarks.

\paragraph*{\textit{Cedar properties and prototype scope}}
Cedar's verified properties (decidability, totality, soundness) hold for the converged state $s^\star$ but not for transient local decisions during partitions---an inherent limitation of any eventually consistent system embedding a single-context policy engine. Mechanized verification of the replay-gate composition would strengthen this claim and is a priority for future work.
The prototype implements the full conceptual model with simplifications: VC validation omits JSON-LD and selective disclosure; key management uses X25519 without a full ceremony protocol; Cedar's full type system is not implemented; and libp2p gossip uses basic flooding. The evaluation uses synthetic workloads on a single Linux x86\_64 machine; E12 simulates multi-node sync in-process; cross-platform determinism awaits multi-platform CI; and the confidentiality layer is experimental. The formal guarantees (Theorems~1--5) are argued but not mechanically verified; a Coq or Isabelle formalization would close the gap between model and implementation.

\section{Conclusion}
\label{sec:conclusion}

\equack{} treats authorization as replicated history.
Data operations, credential changes, and trust updates all go into the
same signed causal event log; replicas merge by set union; and a
deterministic deny-wins replay gate decides which operations survive.
Replicas that have seen the same events therefore compute the same
authorized state, and revoked operations are excluded---even if they
executed while the revoking replica was unreachable.

The prototype evaluation supports this within the limits of a
single-machine, synthetic-workload setup.  One hundred valid orderings
of a 10\,000-operation log all converged to the same state digest;
a revocation workload suppressed every post-revocation write after
reconciliation; multi-issuer conflicts resolved deterministically; and
audit tampering was detected.  Replay throughput reached
45\,000~operations/s on linear chains, and checkpoint restore cut
replay cost by 8--11$\times$.

The workloads are synthetic, the multi-node topology is simulated
in-process, and the confidentiality layer is experimental.  The model
also trades real-time revocation for partition availability: it
guarantees the converged \emph{digital} state, not reversal of
physical-world actions.  Multi-machine evaluation under realistic
network partitions and mechanized verification of the replay-gate
composition are the clear next steps.
