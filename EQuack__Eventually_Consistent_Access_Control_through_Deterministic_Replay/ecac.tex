\author{Aditya Sissodiya, Johan Kristiansson, Eric Chiquito and Ulf Bodin \\
\texttt{Luleå University of Technology, Sweden}}

\IEEEtitleabstractindextext{%
\begin{abstract}
  Intermittently connected multi-stakeholder systems need authorization
  decisions that remain available offline yet converge deterministically
  after partitions heal.  \equack{} is an eventually consistent
  access-control model that records data operations, credential grants
  and revocations, and trust evidence in a single signed causal event
  log.  Replicas merge logs by set union and reconstruct state through
  deterministic replay with a deny-wins gate, so replicas that have
  received the same well-formed events compute the same authorized
  state, operations admitted under subsequently revoked credentials are
  excluded after reconciliation.  In a Rust prototype, 100 valid
  topological orderings of a 10\,000-operation log produced an identical
  state digest; all 500 post-revocation writes in a policy workload were
  suppressed after reconciliation; multi-issuer conflicts resolved
  deterministically; and replay reached 45\,000~operations/s with
  8--11$\times$ faster recovery from checkpoints.
  \end{abstract}

  \section{Introduction}
  \label{sec:introduction}
  Access control in multi-stakeholder environments faces an unresolved tension; devices and services must continue operating when disconnected, yet security-critical decisions, especially credential revocation, must eventually be enforced consistently across all parties. The problem is sharpest in industrial Internet of Things (IoT)~\cite{MISTRY2020106382,liu2020fabriciot}, healthcare~\cite{HASSELGREN2020104040}, and cross-organizational data sharing~\cite{10.1145/3511899,DU2024101878}, where equipment changes hands between manufacturers, operators, and service providers throughout its lifecycle (e.g., a compressor sold by vendor~A, operated by mining company~B, and serviced by maintenance contractor~C, each holding distinct permissions over its control parameters).

  Federated research data sharing~\cite{HASSELGREN2020104040,10.1145/3511899},
  disaster-response coordination, and industrial control share an
  enforcement gap; independent authorities issue and revoke credentials,
  devices operate offline, and permissions evolve while nodes are
  disconnected.  Authorization in these settings is an
  \emph{offline-first} problem; nodes must decide locally, yet the
  system must later reconcile those decisions into a consistent,
  auditable state.

  Consider a factory running hundreds of programmable controllers on a
  restricted network.  Maintenance engineers from different vendors
  receive temporary access to specific machines, but controllers may
  operate for days without reliable
  connectivity~\cite{9551736,s23208535}.  An engineer updates a
  configuration parameter while offline using a credential from their
  employer; hours later, that credential is revoked after a safety
  incident.  When logs from different controllers are synchronized, the
  system must deterministically agree on which parameter changes are
  retained in the converged digital state, and produce an audit trail
  linking each change to the actor, the permission, and the time.
  \equack{} governs which configuration changes are retained in the
  reconciled state; it does not govern whether a controller
  \emph{immediately acts} on a parameter change before reconciliation.
  Applications requiring staged execution of safety-critical commands
  must implement hold-and-confirm logic at the application layer
  (Section~\ref{sec:discussion}).

  Among the approaches surveyed in
  Section~\ref{subsec:gap-analysis}, none provides deterministic
  post-partition reconciliation of fine-grained authorization
  decisions.
  Centralized systems require online coordination; token and
  capability schemes verify offline but cannot reconcile
  conflicting authorization state; Conflict-free Replicated Data Types (CRDTs) converge data but
  not authorization semantics; and ledger-based designs
  require consensus incompatible with disconnected operation
  (Section~\ref{subsec:gap-analysis}).

  \equack{} targets \emph{authorization-rate} workloads:
  access-control decisions issued at the rate of human or
  administrative actions (operations per minute, not per
  millisecond). It is not designed for high-throughput data-plane
  enforcement such as per-packet filtering or real-time telemetry
  ingestion; Section~\ref{sec:discussion} expands on this
  positioning.

  Jacob and Hartenstein~\cite{jacob2024eventuallyac} formalize
  this setting as \emph{Eventually Consistent Access Control}
  (ECAC); replicas autonomously enforce access control from local
  knowledge and reconcile after reconnection. Their model distinguishes
  \emph{causal authorization} (immutable, based on predecessor history) from
  \emph{topological authorization} (mutable, based on the total execution order),
  and provides formal safety and liveness properties, including app state
  confluence under non-monotonic revocations, but does not specify
  how credentials are represented, how trust evidence propagates
  offline, how conflicting grants and revokes from independent
  issuers are resolved, or how fine-grained attribute-based
  policies are evaluated over reconciled history. They provide no
  implementation or experimental evaluation.
  Kleppmann et al.~\cite{kleppmann2019localfirst} independently
  identify access control, particularly revocation, as an
  unsolved problem for local-first CRDT-based software.

  This paper presents \equack{}, an instantiation of the
  ECAC paradigm that closes these mechanism-level gaps. \equack{}
  records data operations, credential grants and revocations, and
  trust evidence in a single signed causal event log. Replicas
  merge logs by set union and reconstruct authorized state through
  deterministic replay with a \emph{deny-wins} gate, a conflict-resolution rule in which any revocation overrides concurrent permits. Authorization is
  expressed through Verifiable Credential (VC)-based time-bounded epochs evaluated by a
  Cedar-inspired policy engine supporting Attribute-Based Access Control (ABAC) and Role-Based Access Control (RBAC) semantics,
  so that fine-grained, scoped, attribute-conditional permissions
  are enforceable offline and reconcilable after partition.

  The novelty is not the individual ingredients; CRDTs, VCs,
  policy engines, but the replay mechanism that composes them into
  a deterministic authorization function over reconciled history,
  providing properties (R2, R3) that---among the systems surveyed
  in Sections~\ref{subsec:gap-analysis}--\ref{subsec:mechanism-comparison}---we
  are not aware of being achieved simultaneously with fine-grained
  policy expressiveness.

  This paper makes the following contributions.
\begin{enumerate}
    \item \textit{ECAC instantiation with fine-grained policy
      support} (Sections~\ref{sec:problem-analysis}--\ref{sec:ecac-model}).
      We instantiate the ECAC paradigm
      with VC-based authorization epochs, in-band trust propagation,
      and Cedar-inspired policy evaluation; mechanisms that ECAC
      defines abstractly but does not specify.

    \item \textit{Deterministic deny-wins reconciliation}
      (Section~\ref{sec:ecac-model}).  We define a causal Directed Acyclic Graph (DAG)
      linearization and replay gate that produces the same authorized
      state on every replica that has received the same events.
      Operations admitted under subsequently revoked credentials are
      excluded after reconciliation.

    \item \textit{Prototype and evaluation}
      (Sections~\ref{sec:ecac-model}--\ref{sec:evaluation}).  \textit{equack-core} is evaluated for convergence, revocation
      enforcement, multi-authority conflict resolution, audit integrity,
      and replay cost.  The experiments confirm identical state digests
      across 100 topological orderings of a 10\,000-operation log,
      suppression of all post-revocation writes, deterministic conflict
      resolution, tamper detection, and replay throughput up to
      45\,000~ops/s with 8--11$\times$ checkpoint speedup.
\end{enumerate}

  Section~\ref{sec:problem-analysis} formalizes the problem and derives requirements.
  Section~\ref{sec:ecac-model} presents the \equack{} model, its Rust prototype
  implementation, and formal claims.
  Section~\ref{sec:evaluation} evaluates it,
  Section~\ref{sec:discussion} discusses trade-offs and positioning, and Section~\ref{sec:conclusion} concludes.

\section{Problem Analysis}\label{sec:problem-analysis}

The offline multi-stakeholder authorization problem arises when systems must (i) continue operating during network partitions, (ii) support multiple independent authorities that can grant and revoke permissions, and (iii) eventually converge to a consistent, auditable authorization state. This section defines the problem context, presents conceptual requirements derived from motivating use cases, and evaluates existing approaches against those requirements.

The requirements and comparisons that follow assume an offline-first, multi-authority setting in which replicas may be disconnected for extended periods, multiple independent organizations issue and revoke credentials, and a computationally bounded adversary may delay, reorder, or partition messages. Section~\ref{subsec:system-model} gives the complete system and threat model; the brief characterization here is sufficient to interpret R1--R6 and the subsequent gap analysis. This setting explains why common fixes fail; online introspection endpoints violate offline-first operation; global consensus violates availability during partition; single-authority trust violates multi-stakeholder governance.

\subsection{The Offline Enforcement Problem}

Consider a set of replicas $R = \{r_1, \ldots, r_n\}$ operated by multiple stakeholders, maintaining shared resources under access control. Each replica may be offline for extended periods, during which it must make local authorization decisions. When replicas reconnect, they must reconcile their local states into a globally consistent outcome.

The core difficulty is not issuing permissions, standard Public Key Infrastructure (PKI) and token mechanisms suffice for that, but ensuring that all replicas eventually agree on \emph{which operations were authorized at the time they occurred}, despite:
\begin{itemize}
    \item \textit{Partition-time divergence}: Different replicas may observe different subsets of grants, revokes, and data operations during a partition.
    \item \textit{Out-of-order delivery}: A revocation may arrive at a replica after operations it should have invalidated.
    \item \textit{Multi-authority conflicts}: Independent issuers may make conflicting decisions (e.g., one grants while another revokes) that must be resolved deterministically.
\end{itemize}

\noindent Figure~\ref{fig:offline-enforcement-problem} illustrates these three challenges and the convergence pipeline that resolves them; after reconnection, replicas merge their event sets, establish a deterministic total order, and replay authorization decisions under a deny-wins policy to reach identical verdicts.

Jacob and Hartenstein~\cite{jacob2024eventuallyac} formalize this setting as ECAC (Section~\ref{sec:introduction}). Their abstract algorithm proves safety and liveness properties, including app state confluence under non-monotonic revocations, but operates at the level of Matrix's power-level-based authorization (LeABAC) rather than credential-based epochs with policy-engine evaluation. \equack{} instantiates their paradigm with concrete mechanisms.

\subsection{System and Threat Model}\label{subsec:system-model}

We consider a set of replicas operated by multiple stakeholders that jointly maintain a replicated log of operations over shared resources. Replicas are deployed in environments with intermittent connectivity (e.g., edge and industrial deployments) and may be offline for extended periods. Each replica locally stores all received operations and materializes application state by deterministic replay.

A \emph{subject} (principal) is an actor that initiates operations (users, services, devices). One or more independent \emph{issuers} create and revoke time-bounded credentials for subjects. Each replica hosts a policy enforcement point (PEP) that admits subject requests only when locally authorized, then emits a signed log event carrying the subject identity and authorization evidence. The event timestamp (HLC) and signature are produced by the replica/PEP, not the subject, preventing forging or backdating under our threat model. All state changes are represented as signed operations in a causal DAG, disseminated via gossip/anti-entropy over an asynchronous network where messages may be delayed, reordered, duplicated, or temporarily dropped.

\equack{} targets security failures arising from \emph{offline enforcement} and \emph{multi-authority governance}. We assume an adversary may (i)~compromise subject credentials and attempt unauthorized operations, (ii)~control the network to delay, reorder, or partition replicas, and (iii)~attempt to tamper with stored history. The adversary cannot forge replica/PEP signing keys or choose HLC timestamps, and is computationally bounded.

The following assumptions scope the guarantees:
\begin{enumerate}
    \item \textit{Offline-first operation.} Replicas may be disconnected for hours to days; solutions requiring continuous connectivity are excluded.
    \item \textit{Multi-authority governance.} Multiple independent organizations issue and revoke credentials; no single root authority controls all permissions.
    \item \textit{Honest-but-buggy administrators.} Issuers follow protocols but may make errors; \equack{} provides safety-by-default semantics.
    \item \textit{Eventual delivery.} Every well-formed operation is eventually delivered to every correct replica, though delivery order may be adversarially influenced.
    \item \textit{Bounded partition duration.} Offline periods are finite; replicas reconcile on reconnection.
    \item \textit{No physical key compromise.} Long-term private keys are not exfiltrated via physical attacks; compromise is modeled at the software/account level.
    \item \textit{Bounded clock skew.} Each replica's physical clock is within a known bound $\epsilon$ of real time. The HLC's physical component inherits this bound; the logical counter and node identifier break ties deterministically regardless of skew. Credential validity intervals $[\mathrm{nb}, \mathrm{na})$ should be configured with margins exceeding $\epsilon$ to avoid boundary ambiguity. Replicas with unbounded clock drift may make incorrect local authorization decisions that are corrected on replay only when the replayed HLC values remain consistent with the true causal order.
\end{enumerate}
\equack{} does not address fully Byzantine collusion, side-channel leakage, cryptographic primitive failures, or metadata privacy; these require additional mechanisms.

\paragraph*{\textit{Scope of guarantees}} \equack{} guarantees convergence and revocation safety for the \emph{digital authorization state}: which operations are retained or excluded in the replicated log after reconciliation. It does \emph{not} prevent a controller from acting on a locally accepted operation before that operation is confirmed or invalidated by replay. In safety-critical deployments where physical actions (actuator commands, firmware updates, safety-parameter writes applied to running equipment) must not execute under stale authorization, the application layer must defer execution until sync confirms the operation's authorization status. \equack{} provides the mechanism for determining that status; it does not prescribe when the application acts on it. The hold-and-confirm pattern described in Section~\ref{sec:discussion} is one such application-layer policy that uses \equack{}'s replay verdicts as input.

\subsection{Conceptual Requirements}\label{subsec:conceptual-requirements}

From the system context and motivating use cases (Section~\ref{sec:introduction}), we derive six conceptual requirements for offline multi-stakeholder access control. These serve as criteria for comparing existing paradigms and guiding \equack{}'s design.

\begin{enumerate}
    \item \textbf{\textit{R1---Offline enforceability.}} Authorization decisions must be enforceable without contacting a central Policy Decision Point (PDP) or ledger; disconnected nodes operate using locally verifiable evidence.
    \item \textbf{\textit{R2---Deterministic post-partition convergence.}} Given the same set of events, all parties must compute the same final authorization state regardless of message arrival order.
    \item \textbf{\textit{R3---Revocation safety under partition.}} Revocations must deterministically invalidate effects in the replicated state after reconciliation, even if the revocation arrives after the operations it should have blocked. Revocation applies to the digital state; irreversible physical-world actions require compensating controls at the application layer.
    \item \textbf{\textit{R4---Tamper-evident accountability.}} The history of grants, revokes, and authorization decisions must form an append-only, verifiable trace.
    \item \textbf{\textit{R5---Decentralized multi-authority issuance.}} Independent issuers must grant and revoke permissions without coordination; verifiers check authenticity offline using locally available evidence.
    \item \textbf{\textit{R6---Confidentiality with post-revocation protection.}} Unauthorized parties should not read protected content; after revocation-triggered key rotation, revoked principals cannot decrypt data created after their access ended.
\end{enumerate}

\noindent\textit{Explicit non-requirement.} We do not address data-plane throughput requirements: \equack{} assumes that authorization decisions are issued at rates compatible with human or administrative workflows (Section~\ref{sec:introduction}), not at per-packet or telemetry-ingest rates.

\subsection{Gap Analysis}\label{subsec:gap-analysis}

Table~\ref{tab:gap-summary} evaluates five dominant approaches against R1--R6. Centralized Identity and Access Management (IAM) couples enforcement to online coordination, failing R1 and R3 under partition~\cite{jacob2024eventuallyac}. Token-based schemes enable offline checks but lack deterministic reconciliation and timely revocation~\cite{rouhani2021distributed}. CRDTs converge data payloads; they do not define authorization semantics or revocation rollback~\cite{shapiro2011crdt,gomes2017_sec}. Blockchains offer total ordering and auditability at the cost of online consensus, which is incompatible with offline enforcement~\cite{8989788,9020182,9446108,8798730,9548956,DIFRANCESCOMAESA201993,s20010282,s22031271}. Capability-based systems~\cite{birgisson2014macaroons,ucan2022,andersen2019wave} verify delegated authority offline but cannot deterministically converge conflicting authorization state or retroactively revoke. Among the paradigms surveyed here, none simultaneously satisfies R1--R6.

Table~\ref{tab:gap-summary} identifies which \emph{properties}
each paradigm satisfies. The deeper gap is mechanism-level; among
the approaches surveyed, none provides
(i)~deterministic reconciliation of authorization history after
    partition,
(ii)~deny-wins revocation semantics that retroactively exclude
    operations admitted under stale credentials,
(iii)~fine-grained policy evaluation (ABAC/RBAC) over the
    reconciled history,
(iv)~in-band propagation of trust and credential-status evidence,
    or
(v)~auditable linkage between credential state and data-state
    replay.
ECAC~\cite{jacob2024eventuallyac} provides the formal framework
for (i) but not the concrete mechanisms for (ii)--(v). \equack{}
addresses these mechanism-level gaps. This claim is based on the
paradigms tabulated in Table~\ref{tab:gap-summary} and the
specific systems compared in
Section~\ref{subsec:mechanism-comparison}; we do not claim
exhaustive coverage of all distributed authorization systems.

\equack{} fills the gap between these approaches, CRDT replication provides data convergence but not authorization convergence~\cite{shapiro2011crdt,gomes2017_sec}; ledgers provide total order but require consensus; token/VC designs support offline checks but lack deterministic reconciliation~\cite{rouhani2021distributed}. \equack{}'s causal DAG linearization plus deny-wins gate makes authorization a reproducible function of the reconciled event set without online consensus.

Capability-based systems---Macaroons~\cite{birgisson2014macaroons}, UCAN~\cite{ucan2022}, Biscuit~\cite{biscuit2023}, WAVE~\cite{andersen2019wave}, and the Willow Protocol~\cite{willow2024}---enable offline verification of delegated authority (R1, R5) but do not address deterministic convergence of conflicting authorization state (R2) or retroactive invalidation of effects authorized under stale evidence (R3). Revocation relies on expiry or propagation, facing the same freshness problem as VC status lists. \equack{}'s contribution is orthogonal; it provides the replay mechanism that deterministically reconciles authorization state after partition healing.

Zanzibar~\cite{zanzibar2019} and its open-source derivatives (e.g., OpenFGA, SpiceDB) provide relationship-based authorization backed by strongly consistent stores, violating R1 by construction. Policy engines such as Open Policy Agent (OPA) and Cedar~\cite{cutler2024cedar} support offline policy \emph{evaluation} but not offline policy \emph{modification} with deterministic convergence. \equack{} sits between these extremes; offline operation with deterministic post-partition convergence, without global consensus.

Matrix's state resolution (v2)~\cite{matrix_stateresv2} is structurally similar; a DAG of events with deterministic state resolution across federated servers, but uses power-level-based authorization rather than CRDT formalism, deny-wins semantics, or VC-based epochs. Jacob and Hartenstein~\cite{jacob2024eventuallyac} formalize Matrix's approach as an ECAC instance; \equack{} builds on this foundation targeting industrial multi-stakeholder settings. \equack{} advances beyond the ECAC abstraction by specifying the credential format (W3C VCs), the conflict resolution semantics (deny-wins epochs), the policy evaluation engine (Cedar ABAC), and the trust propagation mechanism (in-band TRUST events).

The p2panda toolkit~\cite{p2panda2024} is the closest existing system. Its \texttt{p2panda-auth} crate~\cite{p2panda_auth2025} implements a convergent access-control CRDT with a ``strong removal'' resolver over a causal DAG, and its encryption layer~\cite{p2panda_encryption2025} uses Decentralized Continuous Group Key Agreement (DCGKA)-based group keys with forward secrecy via the Signal Double Ratchet. Three differences position \equack{} distinctly; (1)~\equack{} enforces deny-wins replay at the protocol level, the Gate function (Section~\ref{subsec:formal-model}) is mandatory and the materialized state always reflects revocations (Theorem~3), whereas p2panda treats retroactive data invalidation as an application-layer choice; (2)~\equack{} models authorization as VC-based epochs evaluated by a Cedar policy layer with multi-issuer trust propagated in-band, whereas p2panda uses group membership with four permission levels and no policy engine; (3)~\equack{} provides an explicit hash-chained audit trail (Theorem~4), whereas p2panda's tamper evidence is implicit in the signed DAG. p2panda's encryption layer is more mature than \equack{}'s experimental per-tag scheme.

\section{\equack{}: Model and Implementation}
\label{sec:ecac-model}

\equack{} is an Eventually Consistent Access Control model designed for offline-first operation with model-level security and auditability guarantees, supported by prototype evidence.
It combines four elements; (1)~a causal event log with deterministic deny-wins replay, (2)~verifiable credentials for time-bounded authorization epochs, (3)~a hybrid Attribute-Based Access Control (ABAC)/Role-Based Access Control (RBAC) policy model with deny-overrides semantics, and (4)~per-tag encryption with key rotation.

The prototype, \textit{equack-core}\footnote{Source code: \url{https://github.com/adityasissodiya/equack-core}.}, is a Rust library that uses Ed25519 for signatures, BLAKE3 for hashing, XChaCha20-Poly1305 for encryption, and SHA-256 for status list bitstrings. Operations are serialized in Concise Binary Object Representation (CBOR).

Each subsection first describes the conceptual mechanism, then
shows how \textit{equack-core} realizes it. The formal model
(Section~\ref{subsec:formal-model}) is self-contained and does not depend on
implementation choices. We state formal claims for convergence, safety, determinism, audit integrity, and leaver security (post-revocation confidentiality under key rotation). Figure~\ref{fig:ecac-overview} summarizes the four-phase lifecycle (Ingest, Validate, Replay, Query). The key idea is that replicas do not edit shared truth directly; they submit signed evidence to the log first, and deterministic replay decides which evidence becomes state later.

\subsection{Event Types and Causal DAG}

\equack{} maintains an append-only event log structured as a causal DAG, where each event points to its causal predecessors. The DAG encodes happens-before relations, and system state is derived by deterministically sorting and replaying events. A fixed traversal order ensures every replica produces the same sequence.

Concurrent events, those without causal order, are resolved using a composite key; a Hybrid Logical Clock (HLC) value plus a unique operation ID. The HLC combines physical and logical time to preserve causality; when timestamps match, op IDs break ties deterministically. Replay thus becomes a pure function of the event set, guaranteeing identical state across replicas (R2). 

A common point of confusion in offline access control is the overlapping role of causal graphs and Data CRDTs. In \equack{}, the pipeline follows a strict separation of concerns, the DAG and topological sort fix a deterministic, totally-ordered sequence of events to evaluate \textit{authorization and policy validity}. Meanwhile, the CRDT state (e.g., MVReg, OR-Set) strictly handles \textit{data payload convergence}. The policy engine evaluates the strict ordered DAG to build epochs and TrustViews; only authorized DATA events are then handed off to the CRDTs. 

The event DAG acts as a convergent CRDT; merging logs is a set union, commutative, associative, idempotent. Operation structure and DAG mechanics are shown in Figure~\ref{fig:dag-structure}. Each operation carries a signed header with parent links, HLC timestamp, producer key, and typed payload. Replicas accept events opportunistically but only replay them once causal predecessors are present (activated/pending staging), ensuring every replica topologically sorts the same DAG prefix.

Replay follows a deny-wins rule, each event is applied only if the actor held the required permission at that moment; unauthorized events become no-ops. Operations are disseminated via gossip and background synchronization (anti-entropy) using libp2p; this affects liveness but not safety or determinism.

\paragraph*{\textit{\textbf{Implementation}}} The DAG is implemented in \texttt{core::dag}, which stores operations and computes the deterministic topological sort (\texttt{core::replay::topo\_sort}). CRDT state ($\mathrm{MVReg}$, $\mathrm{OR\text{-}Set}$) is managed by \texttt{core::crdt}. Operations are serialized as deterministic CBOR with Ed25519 signatures computed over BLAKE3 hashes of the unsigned header (\texttt{core::crypto}). Gossip dissemination and anti-entropy synchronization use libp2p (\texttt{net::gossip}, \texttt{net::sync}).

\subsection{Authorization Epochs and Credential Lifecycle}

\equack{} manages permissions using verifiable, time-bounded credentials. It adopts the W3C VC format for signed authorization tokens asserting a principal's rights over a specific interval (validFrom--validUntil). Credentials are tamper-evident and self-verifying; replicas independently confirm authenticity and validity~\cite{w3c_vc_2024}.

Revocations use standard VC status lists, cryptographically signed bitstrings indicating whether each credential ID is revoked. A valid credential defines an \emph{authorization epoch}, the interval during which that credential grants a subject a specific permission; expiry or revocation ends it.

Trust evidence is carried in-band as TRUST events (issuer keys and status-list updates) and replayed into a local \emph{TrustView}---the replica's materialized view of all issuer keys and credential-status evidence received so far; anchored by a pinned, cross-signed genesis checkpoint. The log forms a distributed PKI, an offline node can verify at any timestamp that a user held a valid, non-revoked credential without contacting a central authority.

\paragraph*{\textit{\textbf{Genesis checkpoint governance}}}
The pinned genesis checkpoint anchors the trust chain: it
contains the initial set of issuer public keys and is intended to
be cross-signed by all founding stakeholders. In the prototype,
the genesis is created by a single administrative command and
distributed out-of-band; the code path lives in \texttt{core::trustview}
and \texttt{core::trust} and treats the genesis as the first entry
in the replica's \texttt{TrustView}. A production deployment would
require a multi-party ceremony in which each stakeholder
independently verifies the genesis content before signing. Adding
or removing stakeholders after genesis requires a signed
\texttt{GenesisUpdate} event referencing the previous genesis
digest; this mechanism is specified at the event-schema level but
is not yet implemented in the prototype.
The governance protocol for genesis updates (quorum requirements,
key rotation for departing stakeholders, dispute resolution) is
an open design problem that we defer to future work; Section~\ref{sec:discussion}
discusses why it is deliberately out of scope for the present
revision.

During replay, the system checks for a valid credential covering each event's timestamp; if found, the operation applies, otherwise it is skipped.

Trust operations (IssuerKey, IssuerKeyRevoke, StatusListChunk) build the TrustView incrementally during replay. When a VC must be checked, replicas parse the JSON Web Token (JWT), look up the issuer key in TrustView, verify the Ed25519 signature, check status-list revocation, and extract claims, all from in-band log state, with no online PKI lookups.

Unlike the W3C Bitstring Status List specification, which assumes HTTP-accessible status endpoints, \equack{} propagates status-list updates as in-band $\mathsf{TRUST}$ events in the causal DAG. This sidesteps the need for always-online StatusList URLs, status freshness is bounded by event propagation latency rather than HTTP polling intervals. The trade-off is that status information is only as fresh as the latest $\mathsf{TRUST}$ event a replica has received. \equack{}'s design is compatible with W3C Bitstring Status List at the data-format level; deployments with partial connectivity can supplement in-band propagation with periodic HTTP fetches when available.

\paragraph*{\textit{\textbf{Implementation}}} Epoch construction is implemented in \texttt{core::epoch}; VC parsing and Ed25519 signature verification in \texttt{core::vc}. TrustView assembly, materializing issuer keys and status-list evidence from replayed TRUST events, is implemented in \texttt{core::trust} and \texttt{core::trustview}. Status-list bitstrings are verified using SHA-256 checksums.

\subsection{Policy Evaluation and Deny-Wins Replay}

\equack{} uses a policy layer inspired by AWS's Cedar language (open-source ABAC). Policies may permit or forbid actions with a default deny stance; deny-overrides semantics ensure any matching forbid overrides all permits. The framework unifies RBAC and ABAC, roles are principal attributes, and Cedar's entity-attribute model supports hierarchical roles and contextual attributes. Policies are evaluated deterministically at each event's timestamp; identical inputs guarantee identical outcomes across replicas.

A \textsc{REVOKE} event deterministically terminates any authorization epoch opened by a prior or concurrent \textsc{GRANT}, so earlier permits cannot ``win'' due to message reordering. Revocation is not irreversible: a \textsc{GRANT} \emph{after} a \textsc{REVOKE} in the replay order can open a fresh epoch under a different (re-issued) credential. For concurrent \textsc{GRANT}/\textsc{REVOKE} pairs, the deterministic linearization chooses a single order, and deny-wins applies \emph{per epoch}, not per subject.

Concretely (illustrated in Figure~\ref{fig:deny-wins}), consider a concurrent \textsc{GRANT}~$g$ and \textsc{REVOKE}~$r$ for the same subject and scope, linearized as $\mathrm{pos}(r) < \mathrm{pos}(g)$. \textbf{Case A} (re-grant under stale credential): $g$ references the same credential $c_1$ whose status-list entry $r$ closes; the VC check fails at $\mathrm{pos}(g)$ and no epoch opens. \textbf{Case B} (re-grant under fresh credential): $g$ references a new credential $c_2$ not covered by $r$'s status update; the VC check succeeds and a fresh epoch opens at $\mathrm{pos}(g)$. Deny-wins is therefore enforced through two complementary paths -- scope-based epoch closure at the \textsc{REVOKE}'s position and credential-level status-list checks at VC verification time -- without forbidding legitimate re-grants.

Our compliance fixture contains an in-band \texttt{IssuerKey}, credential-and-grant pairs for two writers, 50 concurrent pre-revoke writes from each writer, a merge point, a \texttt{REVOKE} for one writer, and 10 post-revoke writes from each writer. Replay first builds the in-band TrustView, verifies both credentials, and opens epochs for both writers. The later \texttt{REVOKE} closes only writer~B's epoch in the deterministic replay order. As a result, all replicas apply both writers' pre-revoke writes, skip writer~B's post-revoke writes, and continue applying writer~A's writes.

Cedar's formal verification~\cite{cutler2024cedar} establishes decidability, totality, and soundness of authorization evaluation under a single consistent context where all entity data and policies are available and coherent at decision time. In \equack{}'s eventually consistent setting, this assumption holds locally (each replica evaluates deterministically given its local state) but not globally during partitions; two replicas may reach different permit/deny outcomes for the same request if one has received a forbid policy or revocation that the other has not. The deny-wins replay mechanism restores global consistency after convergence, once all replicas hold the same event set, replay re-evaluates every operation under the converged policy set, and the final materialized state $s^\star$ satisfies Cedar's deny-overrides semantics. Cedar's verified properties thus hold for $s^\star$ but not for the transient local decisions made during partitions. Formal verification of the replay mechanism itself, specifically that deterministic linearization plus deny-wins gating preserves the intended authorization invariants is future work and would require extending Cedar's Lean~4 proofs or independent mechanized checking.

The current model uses deny-overrides as the sole combining algorithm, any matching \texttt{forbid} policy overrides all \texttt{permit} policies. This is a deliberate safe default for multi-authority settings where no single authority should be able to override another authority's denial. Priority-based or temporal-precedence combining (e.g., last-writer-wins for policy updates) would require extending the epoch model to track policy precedence across authorities; we leave this for future work.

Entity attributes used in Cedar policy evaluation (e.g., role assignments, group memberships) may be stale during partitions, a replica that has not yet received an attribute-update event evaluates policies against outdated entity state. Because attribute changes are events in the causal DAG, they are replayed deterministically on convergence, and the final state $s^\star$ reflects the correct attribute values. Transient local decisions based on stale attributes are corrected during replay.

\paragraph*{\textit{\textbf{Implemented Cedar subset}}}
The prototype's evaluator is purpose-built (\texttt{crates/core/src/policy.rs}, $\sim$1{,}000 LOC including the epoch index, the deny-wins gate, and reason classification). It is not the upstream Cedar SDK. We therefore enumerate exactly which Cedar concepts the prototype implements and which it does not, so reviewers can match Listing~\ref{lst:cedar-fleet} against the actual decision procedure:

\begin{itemize}
  \item \textbf{Implemented:} \texttt{permit}/\texttt{forbid} effects with deny-overrides semantics; principal binding by Ed25519 public-key entity (\texttt{author\_pk}); action discrimination across the three internal actions \texttt{SetField}/\texttt{SetAdd}/\texttt{SetRem} parsed from the data-op key prefix; static resource-tag inference (\texttt{tags\_for}); scope intersection between credential scope and resource tags (\texttt{is\_disjoint} check on \texttt{TagSet}); time-bound enforcement against the op's HLC via the credential's \texttt{nbf}/\texttt{exp} claims; topo-position bounds via the epoch's \texttt{[start\_pos, end\_pos)} window; deny classification with reason codes (\texttt{RevokedCred}, \texttt{ExpiredCred}, \texttt{OutOfScope}, \texttt{GenericDeny}) returned to the audit hook; and a fixed role $\rightarrow$ permission table for the \texttt{editor} and \texttt{issuer\_admin} roles.
  \item \textbf{Not implemented:} Cedar's full type system and schema validation; \texttt{unless} clauses; partial evaluation; the \texttt{when} clause grammar (boolean expressions over \texttt{context} and \texttt{resource} attributes such as Listing~\ref{lst:cedar-fleet}'s \texttt{context.shift == "day"} and \texttt{resource.safety\_level < 3} are interpreted at the policy-author level and lowered into static tag membership before they reach the evaluator); arbitrary entity hierarchies beyond the role/tag pairs registered in \texttt{role\_perms}/\texttt{tags\_for}; policy templates; and Cedar's Lean~4-verified soundness/decidability proofs (we re-derive the relevant deny-wins property informally in Section~\ref{subsec:main-theorems}, Theorem~3).
\end{itemize}

The prototype scope is therefore close to ABAC-with-static-tags rather than full Cedar. Migrating to the upstream \texttt{cedar-policy} Rust SDK is a planned drop-in replacement (Section~\ref{sec:discussion}, ``Cedar integration roadmap''); the current evaluator predates the Rust SDK's stabilisation and is retained because it is small, deterministic, and easy to wire into the deny-wins gate. The two design knobs that would change in a Cedar-SDK port are (1)~replacing \texttt{role\_perms}/\texttt{tags\_for} with a Cedar policy set parsed at boot, and (2)~replacing the bespoke \texttt{is\_permitted\_at\_pos\_with\_reason} call with a Cedar \texttt{Authorizer} call whose \texttt{Decision} we map onto the existing \texttt{DenyDetail} enum.

\subsection{Tamper-Evident Audit Trail}

During replay, each replica emits a signed \emph{audit record} per DATA operation indicating applied/skipped status and reason. Records are linked in an append-only hash chain; any inconsistency is detectable relative to a pinned chain head digest (e.g., cross-signed checkpoint). Because replay is deterministic, independent replicas with the same event set compute identical audit traces and can validate each other after reconciliation (R4).

\paragraph*{\textit{\textbf{Implementation}}} Audit chain management is implemented in \texttt{store::audit}. The \texttt{replay\_full\_with\_audit} entry point accepts an \texttt{AuditHook} trait object, emitting signed records during replay. Checkpoints (state snapshot plus topological watermark) are managed by \texttt{store::checkpoint} backed by RocksDB.

\subsection{Confidentiality and Key Rotation}

\equack{} supports an optional per-tag payload encryption layer. Each DATA operation carries ciphertext $c := \mathsf{Enc}_{K_{\tau}^{(v)}}(m)$ with key version $v$ (XChaCha20-Poly1305). Ciphertext is replicated for convergence; access control is enforced at \emph{query time} via the deny-wins gate, with unauthorized subjects receiving a redacted view. On revocation, a KeyRotate event advances to $K_{\tau}^{(v+1)}$, distributed in-band via KeyGrant records encrypted to authorized subjects' public keys. This provides \emph{leaver security}; revoked subjects lacking $K_{\tau}^{(v+1)}$ cannot decrypt post-rotation ciphertexts. This component is experimental; we treat confidentiality results as indicative.

\paragraph*{\textit{\textbf{Implementation}}} Encryption and decryption use XChaCha20-Poly1305 in \texttt{core::crypto}. Per-tag key versioning and KeyGrant/KeyRotate handling use X25519 key agreement without a full ceremony protocol. Tag keys are persisted by \texttt{store::put\_tag\_key} / \texttt{store::get\_tag\_key}.

\subsection{Formal Model and Guarantees}\label{subsec:formal-model}

Let $\mathsf{Subj}$ be subjects (principals), $\mathsf{Res}$ resources, $\mathsf{Act}$ actions, and $\mathsf{Attr}$ attribute valuations.
Physical time is measured in milliseconds as $(T_{\mathrm{ms}}=\mathbb{N}, \le)$.
Let $\mathsf{NodeId}$ be replica identifiers, and let hybrid logical clock timestamps range over
$T_{\mathrm{hlc}} := T_{\mathrm{ms}} \times \mathbb{N} \times \mathsf{NodeId}$ ordered lexicographically.
Let $\mathrm{phys} : T_{\mathrm{hlc}} \to T_{\mathrm{ms}}$ project the physical millisecond component.
We use $T_{\mathrm{ms}}$ for credential validity intervals and status freshness, and $T_{\mathrm{hlc}}$ (via lexicographic order) only for deterministic linearization.
Let $\mathsf{H}$ be a collision-resistant hash and $\mathsf{Sig}_k(\cdot)$ a secure signature scheme.


%\noindent\textbf{Verifiable credentials and trust.}
A verifiable credential (VC) is a tuple
\[
  c = \langle s,\, \alpha,\, V,\, \mathrm{nb},\, \mathrm{na},\, \sigma_I\rangle
\]
where $s\!\in\!\mathsf{Subj}$, $\alpha\!\in\!\mathsf{Attr}$ (roles/scopes), $V$ issuer identifier,
and $\mathrm{nb},\mathrm{na}\!\in\!T_{\mathrm{ms}}$ are not-before/after bounds.
$\sigma_I$ is a signature under issuer key $K_V$.
A status list is a signed, time-versioned snapshot published by issuer $V$ that maps credential identifiers to $\{\mathsf{good},\mathsf{revoked}\}$.
We model status evidence as a sequence of $\mathsf{TRUST}$ events that carry status-list updates (or chunks).
Let $\mathsf{SL}^{i}_{V}$ denote the status map materialized by replaying all $\mathsf{TRUST}$ events with $\mathrm{pos}(e) \le i$ and taking the latest published status for each credential id.
When validating a credential for an event $e$, we query $\mathsf{SL}^{\mathrm{pos}(e)}_{\mathrm{iss}(c)}(\mathrm{id}(c))$.
A \emph{TrustView} event carries (i) issuer keys $\{K_V\}$ and (ii) status-list updates, recorded in-band so replicas can validate VCs offline using the latest status snapshot up to position $\mathrm{pos}(e)$ in the replay order.



Events are partitioned as
\[
  \mathsf{Ev} \;=\; \mathsf{DATA}\;\uplus\;\mathsf{GRANT}\;\uplus\;\mathsf{REVOKE}\;\uplus\;\mathsf{TRUST}\;\uplus\;\mathsf{KEY}.
\]
The subset $\mathsf{KEY}$ contains key-management events (e.g., \textsc{KeyRotate}, \textsc{KeyGrant}) that update per-tag key state used for query-time decryption.
Each event $e\in\mathsf{Ev}$ has an \emph{unsigned} header $h_0(e)$ containing domain fields, causal parents $\mathsf{par}(e)\subseteq\mathsf{Ev}$, a hybrid logical clock $\mathrm{hlc}(e)\in T_{\mathrm{hlc}}$ assigned by the producing replica, the producer public key $\mathrm{pk}_r$, and the payload (or a payload hash). We define the event identifier as
\[
  \mathsf{op\_id}(e) := \mathsf{H}(\mathrm{domain}\,\|\,\mathrm{CBOR}(h_0(e))).
\]
The producing replica signs this identifier:
\[
  \sigma_e := \mathrm{Sig}_{\mathrm{sk}_r}(\mathsf{op\_id}(e)).
\]
Signing $\mathsf{op\_id}(e)$ is sufficient because $\mathsf{op\_id}(e)$ commits to the canonical bytes of $h_0(e)$ under collision resistance of $\mathsf{H}$.
The full header is $h(e) := (h_0(e), \sigma_e)$. This avoids any circularity; $\mathsf{op\_id}(e)$ is derived from the unsigned header, and $\sigma_e$ covers $\mathsf{op\_id}(e)$.
The \emph{signed, hash-linked op log} is an append-only bag of events with edges $e'\!\to\! e$ for $e'\in\mathsf{par}(e)$.
The induced graph $G=(\mathsf{Ev},E)$ must be a DAG; this causal DAG defines the partial order $\prec$ as the transitive closure of $E$.



Let the deterministic key be
\[
  k(e) := \big(\mathrm{hlc}(e),\mathsf{op\_id}(e)\big),
\]
ordered lexicographically. Define the replay sequence $\langle e\rangle_{\triangleleft}$ as the output of Algorithm~\ref{alg:linearize} applied to the causal DAG $G$. We then define $e_i \triangleleft e_j$ iff $e_i$ appears before $e_j$ in $\langle e\rangle_{\triangleleft}$.

\begin{algorithm}[!htbp]
\caption{Deterministic linearization of a causal DAG}
\label{alg:linearize}
\begin{algorithmic}[1]
\Require DAG $G=(V,E)$, key $k(e)=(\mathrm{hlc}(e),\mathsf{op\_id}(e))$
\Ensure Sequence $L$ that is a deterministic linear extension of $\prec$
\State $\mathrm{indeg}[v] \gets |\{(u,v)\in E\}|$ for all $v\in V$
\State $Q \gets$ min-priority queue ordered by $k(\cdot)$
\ForAll{$v\in V$ with $\mathrm{indeg}[v]=0$}
  \State $Q.\mathrm{push}(v)$
\EndFor
\State $L \gets [\ ]$
\While{$Q$ not empty}
  \State $v \gets Q.\mathrm{pop\_min}()$
  \State append $v$ to $L$
  \ForAll{$(v,w)\in E$}
    \State $\mathrm{indeg}[w] \gets \mathrm{indeg}[w]-1$
    \If{$\mathrm{indeg}[w]=0$}
      \State $Q.\mathrm{push}(w)$
    \EndIf
  \EndFor
\EndWhile
\State \Return $L$
\end{algorithmic}
\end{algorithm}

\noindent Let $\langle e \rangle_{\triangleleft} = [e_0,\dots,e_{n-1}]$ be the output of Algorithm~\ref{alg:linearize}.
Define the deterministic replay position $\mathrm{pos}(e_i) := i$.

If two events $e_A$ and $e_B$ are concurrent and both are ready (zero indegree) at some step, Algorithm~\ref{alg:linearize} selects the smaller key, if $\mathrm{hlc}(e_A)<\mathrm{hlc}(e_B)$ then $e_A$ is chosen first; if equal, the smaller $\mathsf{op\_id}$ is chosen.




Epochs are computed deterministically by replaying non-\textsc{DATA} events in $\langle e \rangle_{\triangleleft}$.
A \textsc{GRANT} event $g$ at position $p_g := \mathrm{pos}(g)$ that references a valid credential opens an epoch record
$\mathsf{ep} = \langle s,a,\mathrm{scope}, p_{\mathrm{start}}{=}p_g, p_{\mathrm{end}}{=}\infty, \mathrm{nb}, \mathrm{na} \rangle$.
A \textsc{REVOKE} event $r$ at position $p_r := \mathrm{pos}(r)$ closes matching open epochs by setting
$p_{\mathrm{end}} := \min(p_{\mathrm{end}}, p_r)$.
We write $\mathsf{ep}\in \mathsf{Epochs}(s,a)$ for epochs associated with subject $s$ and action $a$.


For subject $s$, scope $a\!\in\!\mathsf{Act}$, and event $e$, define the authorization predicate
\[
\begin{aligned}
\mathsf{Auth}(s,a,e) := \exists c\ \text{s.t.}\ & \mathrm{phys}(\mathrm{hlc}(e)) \in [\mathrm{nb},\mathrm{na})\ \land\ c \text{ grants }(s,a)\\
& \land\ \mathsf{SL}^{\mathrm{pos}(e)}_{\mathrm{iss}(c)}(\mathrm{id}(c))=\mathsf{good}.
\end{aligned}
\]
Status-list Time-to-Live (TTL) is a freshness hint for operational policy; our determinism guarantees assume replicas replay the same TrustView history, while deployments may additionally deny decisions when status evidence is older than a configured threshold.

\paragraph*{\textit{Clock skew and validity boundaries}} The authorization predicate $\mathsf{Auth}(s,a,e)$ evaluates credential validity using $\mathrm{phys}(\mathrm{hlc}(e))$, the physical millisecond component of the event's HLC, which is recorded by the producing replica at event-creation time. Under the bounded clock skew assumption (Section~\ref{subsec:system-model}, item~7), this recorded timestamp is within $\epsilon$ of real time. Consequently, an event close to a credential's $\mathrm{nb}$ or $\mathrm{na}$ boundary may be classified differently than intended: a replica whose clock is skewed by $\delta \le \epsilon$ from real time records a physical component off by $\delta$, so events within $\epsilon$ of a boundary may be accepted on one replica and rejected on another during the partition. Setting credential validity margins larger than $\epsilon$ eliminates this ambiguity for events that are well inside the validity interval. Crucially, deterministic convergence (Theorem~1) is \emph{unaffected} by skew: after reconciliation, all replicas evaluate the same $\mathrm{hlc}(e)$ against the same credential bounds and produce the same verdict. Determinism depends on the HLC values recorded in the log, not on their accuracy relative to real time; only the \emph{intended} semantic correspondence between $\mathrm{phys}(\mathrm{hlc}(e))$ and real wall-clock time degrades with skew.



Let $\mathcal{P}$ denote the Cedar policy, modeled as a total function
\[
  \mathcal{P} : \mathsf{Subj}\times\mathsf{Act}\times\mathsf{Res}\times\mathsf{Attr}\to\{\mathsf{allow},\mathsf{deny}\},
\]
with \emph{deny-overrides} semantics.
For a $\mathsf{DATA}$ event $e$ with tuple $(s,a,r,\alpha_e)$, define the \emph{gate function}---the predicate that determines whether a data event is authorized at its position in the replay order:
\[
\begin{aligned}
\mathsf{Gate}(e) :=\;&
\Big(
    \exists\, \mathsf{ep} \in \mathsf{Epochs}(s,a)
    \;\text{s.t.}\; \\
&\qquad
    \mathrm{pos}(e) \in [p_{\mathrm{start}}, p_{\mathrm{end}})
    \;\wedge \\
&\qquad
    \mathrm{phys}(\mathrm{hlc}(e))
    \in [\mathrm{nb}, \mathrm{na})
\Big) \\
&\wedge\;
\mathcal{P}(s,a,r,\alpha_e) = \mathsf{allow}.
\end{aligned}
\]
If \emph{any} deny applies, $\mathcal{P}$ returns $\mathsf{deny}$ and the DATA op is skipped (deny-wins).



Let $(\mathcal{S},\bullet)$ be the state space with a deterministic update function
$U:\mathcal{S}\times\mathsf{DATA}\to\mathcal{S}$.
We assume the DATA payloads target CRDTs such as $\mathrm{MVReg}$ and $\mathrm{OR\!-\!Set}$ whose effect under the same total order is deterministic.
Given initial state $s_0$, define \emph{deterministic replay} over the sequence $\langle e\rangle_{\triangleleft}$:
\[
\begin{aligned}
\mathsf{Replay}(s_0, \langle e\rangle_{\triangleleft}) :=\ &
\mathrm{fold}\Big(s_0,\ \lambda(s,e).\\
& \begin{cases}
  U(s,e) & \text{if } e\in\mathsf{DATA}\ \land\ \mathsf{Gate}(e)\\
  s      & \text{otherwise}
  \end{cases}\Big).
\end{aligned}
\]
The \emph{materialized state} (the computed state after replaying all events) is $s^\star := \mathsf{Replay}(s_0,\langle e\rangle_{\triangleleft})$.
Checkpoints store $(\mathsf{H}(s^\star),\mathrm{meta})$ periodically together with the append-only event store.


Define the audit stream as a sequence of records
\[
\begin{aligned}
\mathsf{Aud}(i) &= \big\langle e_i,\ \mathsf{Gate}(e_i),\ \mathsf{H}(s_i)\big\rangle,\\
\mathrm{ch}_0 &:= \bot,\quad
\mathrm{ch}_{i+1} := \mathsf{H}\!\big(\mathrm{ch}_i \,\|\, \mathsf{Aud}(i)\big),
\end{aligned}
\]
signed by the auditing replica (producer) to yield a tamper-evident chain $(\mathrm{ch}_i)$ of applied/skipped decisions.


Let $\mathsf{Tag}$ denote data tags; each tag $\tau$ has a versioned symmetric key $K_{\tau}^{(v)}$. A DATA payload $m$ is stored as $(c,v)$ where $c := \mathsf{Enc}_{K_{\tau}^{(v)}}(m)$. Ciphertexts are replayed and replicated for convergence; authorization controls \emph{decryption} at query time. On a KeyRotate event for $\tau$ at time $t_r$, the current key version advances to $v{+}1$ and subsequent writes use $v{+}1$; KeyGrant records distribute $K_{\tau}^{(v+1)}$ only to subjects authorized to read $\tau$ after $t_r$.

Events are disseminated via secure anti-entropy gossip; missing parents are fetched until $G$ is complete.
This affects \emph{when} a replica can replay but not \emph{what} $s^\star$ is, once the same set of events is present.

\paragraph*{\textit{Pre-convergence safety window}} Let $r$ be a revocation event created at physical time $t_r$ on some replica. Define the \emph{safety window} $W(r)$ as the interval $[t_r, t_{\text{heal}}]$ where $t_{\text{heal}}$ is the earliest time at which all correct replicas have received $r$ via anti-entropy. During $W(r)$, replicas that have not yet received $r$ may accept operations from the revoked principal under stale authorization evidence. Under the bounded partition assumption (Section~\ref{subsec:system-model}, item~5), $W(r) \leq \Delta_{\text{partition}} + \Delta_{\text{propagation}}$ is finite.

\equack{} does not prevent tentative authorization during $W(r)$; it guarantees that the materialized state $s^\star$ computed after reconciliation reflects $r$. Specifically, any $\mathsf{DATA}$ event $e$ from the revoked subject with $\mathrm{pos}(e) \geq \mathrm{pos}(r)$ in the deterministic order $\triangleleft$ is skipped by the deny-wins gate, regardless of which replica originally accepted $e$. This provides \emph{retroactive revocation safety}, the final digital state is consistent with the revocation, even though individual replicas may have temporarily acted on stale evidence. In contrast, Zanzibar's Zookie mechanism~\cite{zanzibar2019} provides real-time revocation at the cost of availability during partition (consistent with Consistency--Availability--Partition tolerance (CAP) theorem constraints). \equack{} trades real-time revocation for availability, repairing digital-state safety on merge (physical-world consequences are the application's responsibility; see Section~\ref{sec:discussion}).

\subsection{Main Theorems}\label{subsec:main-theorems}

R1--R6 in Section~\ref{subsec:conceptual-requirements} are \emph{system-level requirements} (deployability and security objectives) for offline-first, multi-stakeholder settings.
The theorems below are \emph{formal guarantees} about \equack{}'s deterministic replay semantics and audit/confidentiality mechanisms under the model assumptions.
The gap analysis (Table~\ref{tab:gap-summary}) maps R1--R6 to paradigm coverage; the theorems below formalize the guarantees, and the evaluation (Section~\ref{sec:evaluation}) provides empirical evidence.

The guarantees below are model-level properties of the deterministic replay semantics. They hold under the assumptions stated in Section~\ref{subsec:system-model}, assuming correct implementation of the replay and gate logic. The prototype (Section~\ref{sec:evaluation}) provides empirical evidence, but the replay-gate composition has not been mechanically verified; the guarantees should therefore be understood as formal claims supported by proof sketches and prototype testing, not as end-to-end verified system properties. Mechanized verification of the replay-gate composition is a priority for future work.

\noindent\textbf{\textit{Theorem~1 (Determinism).}}
\textit{For any DAG $G$ and initial state $s_0$, the order $\triangleleft$ is unique and
$\mathsf{Replay}(s_0,\langle e\rangle_{\triangleleft})$ yields a unique $s^\star$.}


\textit{Proof sketch.} Two-step induction. (i) On DAG structure: Algorithm~\ref{alg:linearize} is Kahn's topological sort with a deterministic min-heap keyed by $(\text{HLC},\text{key id})$ over the ready set. By induction on $|G|$, at each step the heap admits a unique minimum (keys are total), so removing it leaves a smaller DAG with a unique extension; concatenation yields a unique linear extension $\langle e\rangle_{\triangleleft}$ for $G$. (ii) On fold position: $\mathsf{Replay}$ is a left fold $s_{i+1}=U(s_i,e_i)$ where $U$ is a pure, deterministic function (CRDT merge plus gate-guarded effect). By induction on $i$, $s_i$ depends only on $s_0$ and $e_0,\dots,e_{i-1}$, so the final $s^\star=s_n$ is unique. \qed


\noindent\textbf{\textit{Theorem~2 (Convergence).}}
\textit{Fix bootstrap anchors $B$ consisting of a pinned, cross-signed genesis checkpoint digest (and the stakeholder public keys used to validate it).
Let two replicas start from the same anchors $B$ and ingest the same multiset of well-formed events (including $\mathsf{TRUST}$ and $\mathsf{KEY}$ events).
Then their materialized states are byte-identical: $s^\star_1 = s^\star_2$.}


\textit{Proof sketch.} Follows from Theorem~1 plus the observation that identical bootstrap anchors and identical event multisets produce identical DAGs: each replica's parent set is recovered from the events themselves, and set union is commutative and idempotent, so $G_1=G_2$. By Theorem~1, both replicas obtain the same linearization $\langle e\rangle_{\triangleleft}$, the same $\mathrm{pos}(\cdot)$ assignments, the same epoch construction, and the same deny-wins gate decisions; applying the same ordered sequence of authorized \textsc{DATA} updates under deterministic $U$ yields byte-identical $s^\star$. \qed


\noindent\textbf{\textit{Theorem~3 (Policy-Safety, deny-wins).}}
\textit{Every effect in $s^\star$ originates from a $\mathsf{DATA}$ event $e$ such that $\mathsf{Gate}(e)$ held at position $\mathrm{pos}(e)$.
If any deny applies (policy or missing epoch), $e$ has no effect.}


\textit{Proof sketch.} Induction on position $i$. At each step, $\mathsf{Gate}(e_i)$ is evaluated against the epoch table built from events $e_0,\dots,e_{i-1}$ (Theorem~1 fixes this prefix uniquely). If any \textsc{REVOKE} at position $j<i$ closed the relevant epoch, then $p_{\text{end}}\le j<i$, so $\mathrm{pos}(e_i)\notin[p_{\text{start}},p_{\text{end}})$ and the gate returns $\mathsf{deny}$; the deny-wins combinator (Section~\ref{subsec:formal-model}) ensures this overrides any concurrent allow. By definition of $\mathsf{Replay}$, $e_i$ then produces no state transition. Hence every effect that survives in $s^\star$ originates from a $\mathsf{DATA}$ event whose gate held at $\mathrm{pos}(e)$. \qed


\noindent\textbf{\textit{Theorem~4 (Audit tamper-evidence).}}
\textit{Let $\mathrm{ch}_N$ be a pinned audit-chain head digest for the first $N$ records (obtained out-of-band or via cross-signing). If the audit chain verifies and the computed head equals $\mathrm{ch}_N$, then any modification to the first $N$ audit records (deletion, insertion, reordering, or bit-flip) is detectable with overwhelming probability.}


The head digest is a collision-resistant hash of the prefix. Any change to the prefix changes the computed head unless the adversary finds a collision/second preimage or forges a signature.


\noindent\textbf{\textit{Theorem~5 (Leaver security).}}
\textit{Fix a tag $\tau$ and a rotation event at time $t_r$ that advances to key version $v{+}1$. Assume indistinguishability under chosen-plaintext attack (IND-CPA) encryption and that $K_{\tau}^{(v+1)}$ is delivered only to subjects authorized for $\mathrm{read}_\tau$ at times $\ge t_r$. Then any subject not authorized for $\mathrm{read}_\tau$ at times $\ge t_r$ cannot decrypt ciphertexts produced under version $v{+}1$, except with negligible probability.}


\textit{Proof sketch.} By assumption, unauthorized subjects do not obtain $K_{\tau}^{(v+1)}$. Under IND-CPA security, ciphertexts under an unknown key reveal no information about plaintext beyond negligible probability. \qed

\textit{Scope and gap.} This claim is conditional on the key-distribution assumption: that $K_{\tau}^{(v+1)}$ reaches exactly the post-rotation authorized set. Closing this gap requires a group-key-agreement ceremony (e.g., a Messaging Layer Security style protocol) that the paradigm does not itself supply. The \equack{} prototype models distribution with X25519 point-to-point envelopes per authorized subject (Section~\ref{sec:evaluation}); this is sufficient to measure rotation cost (E14) but does not constitute a deployable ceremony. A production deployment would substitute a group-key-agreement protocol; integrating one and re-stating Theorem~5 with its concrete adversary model is future work (Section~\ref{subsec:limitations}).

\section{Evaluation}
\label{sec:evaluation}

We evaluate \equack{} through deterministic experiments that provide
empirical evidence for the main theorems (Theorems~1--5,
Section~\ref{subsec:main-theorems}) and quantify replay cost.
Table~\ref{tab:experiment-summary} maps each experiment to the

\subsection{Methodology}\label{subsec:methodology}

Experiments ran on a single Linux x86\_64 workstation using Rust 1.85 in release mode. Multi-node convergence (E12) uses in-process deterministic simulation; partition behavior is validated via offline log merges. State digests are computed as $\mathrm{blake3}(\mathrm{DOMAIN}\,\|\,\mathrm{canon}(\textit{state}))$, where $\mathrm{canon}(\cdot)$ is a deterministic byte serialization with stable key ordering.

We use three synthetic workloads (Table~\ref{tab:workloads}) and two policy-focused CBOR logs. The \emph{hb-chain} workload is a single writer appending sequentially to one key, giving a baseline with no concurrency or policy overhead. The \emph{concurrent} workload has eight writers modifying the same key, stressing the MVReg CRDT's conflict resolution. The \emph{offline-revocation} workload mixes writes with mid-workload revocations, forcing epoch recalculation during replay. Two additional logs exercise deny-wins behavior; \texttt{e3-policy.cbor} (grant, 500 writes, revoke, 500 writes) and \texttt{e4-policy.cbor} (multi-issuer grants, revoke, post-revoke writes). For the partition scenario, logs \texttt{partition-A.cbor} and \texttt{partition-B.cbor} were generated and merged to model revocation propagation on reconnection.

\subsection{Correctness}\label{subsec:correctness}

\paragraph*{\textbf{\textit{Convergence (E1; T1, T2)}}}
A 10{,}000-op concurrent-workload log was permuted into 100 valid orderings (respecting causal dependencies) and each was replayed independently. All 100 produced the same digest (\texttt{\small cbab89ee\ldots b7f493b2}), with mean replay time 615~ms ($\sigma=18$~ms). Delivery order does not affect the final authorized state; a prerequisite for safe offline operation across independently administered replicas.

\paragraph*{\textbf{\textit{Deny-wins revocation (E3; T3)}}}
User $U$ was granted write access, performed 1{,}000 writes, and had the credential revoked after write~500. The 1{,}004-op log (1{,}000 Data + IssuerKey, Credential, Grant, Revoke) was replayed; the first 500 writes applied, the remaining 500 skipped. Digest: \texttt{\small 8da8b49e\ldots 46633fbf}. An engineer whose credential is revoked after a safety incident has all post-revocation parameter changes excluded from the converged state, regardless of which controllers received the revocation first.

\paragraph*{\textbf{\textit{Revocation under partition (E11; T2, T3)}}}
In partition~A a user is revoked; partition~B, which has not yet received the revocation, contains further writes from the same user under stale evidence. After merging both logs and replaying, the 50 pre-revocation data writes (from both partitions) were applied and all 10 post-revocation writes from partition~B were skipped. A revocation issued on one side of a partition retroactively invalidates stale writes accepted on the other side.

\paragraph*{\textbf{\textit{Multi-authority conflict (E4; T1, T3)}}}
Three issuers ($I_1$, $I_2$, $I_3$) with overlapping authority grant user $U$ the ``editor'' role at $t_1$, $t_2$; $I_3$ revokes at $t_3>t_2$. The resulting 128-op log was injected into 10 replicas in random orders. All converged to the same digest (\texttt{\small f2fbb024\ldots 1103c2b2}) writes during $[t_1,t_3)$ accepted, writes after $t_3$ skipped.

\paragraph*{\textbf{\textit{Audit integrity (E5; T4)}}}
An audited log was generated and the chain head digest $\mathrm{ch}_N$ was recorded. Three tamper modes were applied; (1)~a byte flip in one record's payload, (2)~deletion of one record from the pinned prefix, and (3)~a swap of two adjacent records. Chain verification against $\mathrm{ch}_N$ succeeded on the untampered log and failed on all three mutations, detecting each at the exact tampered record. Any party can independently verify that no authorization decisions were altered after the fact.

\paragraph*{\textbf{\textit{Replay determinism regression (E2; T1)}}}
A 127-op CBOR fixture exercises all event categories (Data, Grant, Revoke, Trust) and includes true concurrency. A Continuous Integration (CI) test replays it on one Linux x86\_64 target and asserts the final digest equals a hardcoded constant (\texttt{\small 4ea9684a\ldots bdc2b9}, blake3). Any replay-engine change that alters the digest breaks the build; this serves as a regression guard on the deterministic linearization and gate logic rather than as a cross-platform validation. We additionally verified that the Raspberry Pi 4 (aarch64) run of the IoT-fleet workload (E17) produced a digest byte-identical to the x86\_64 reference for the same seed, confirming cross-platform determinism across ISAs.

\paragraph*{\textbf{\textit{Multi-node sync (E12; T2, T3)}}}
In an in-process simulation, three nodes each generated 1{,}000 ops independently, then synced; all converged. A second scenario partitioned node~C while A issued a Revoke targeting C. After healing and full sync, all three nodes held the same digest, with C's post-revoke ops gated out (Table~\ref{tab:e12-sync}). This demonstrates end-to-end deny-wins enforcement across independently operating nodes.

\paragraph*{\textbf{\textit{Asymptotic cost}}}
Let $n = |\mathsf{Ev}|$ be the number of events and $m = |E|$ the number of causal DAG edges. Algorithm~\ref{alg:linearize} (deterministic linearization) runs in $O(n \log n + m)$: each event is inserted into and extracted from the min-heap exactly once ($O(n \log n)$), and each edge is traversed once ($O(m)$). Replay is a single pass over the linearized sequence: $O(n)$ gate evaluations, each requiring epoch lookup ($O(\log k)$ for $k$ concurrently open epochs in a subject's scope) and Cedar-inspired policy evaluation ($O(p)$ for $p$ policy rules). TrustView growth is $O(t)$ where $t$ is the number of TRUST events (issuer keys and status-list chunks); in practice $t \ll n$. Checkpoint restore is $O(1)$ for the state snapshot plus $O(n - n_{\text{ckpt}})$ for replaying the suffix, matching the near-constant-factor speedup observed in Table~\ref{tab:e6-replay}. Storage is $O(n)$ for the append-only log (approximately 430~bytes/op in CBOR, Table~\ref{tab:e8-storage}) plus $O(n)$ for RocksDB indices. Peak memory during replay is dominated by the in-memory DAG ($O(n + m)$) and CRDT state ($O(n)$ in the worst case; sub-linear when multi-value registers collapse conflicting writes).

\paragraph*{\textbf{\textit{Adversarial resilience (E18, E19; T1--T4)}}}
E16 and E17 measure benign-fault behaviour: nodes go offline, recover, and merge cleanly. The reviewer (W3, 10\,April 2026) asks for adversarial scenarios in which an attacker actively tries to subvert the gate. We add two experiments, both run by the dedicated harness \texttt{bench\_adversarial} (\texttt{crates/core/src/bin/bench\_adversarial.rs}), which collapses each scenario to an applied/skipped pair read from the metrics registry and asserts the expected outcome inside the bench so any regression fails CI.

E18 covers \emph{credential compromise}. A subject with a valid grant emits $50$ legitimate writes; the issuer then revokes the grant; the subject (assumed compromised) keeps writing $50$ further data ops with HLCs strictly later than the revoke. After the merged log is replayed, the gate skips every one of the $50$ post-revoke writes ($50$ applied, $50$ skipped). E18 also includes a \emph{forged grant} sub-scenario: a rogue issuer mints a grant for the attacker referencing a credential whose \texttt{iss} claim points at an issuer that has no \texttt{IssuerKey} on the log. \texttt{TrustView::select\_key} returns \texttt{None}, the VC fails verification, no epoch opens, and the $10$ data writes the attacker subsequently emits are all denied at the gate.

E19 covers \emph{forged event injection}. (a)~\textbf{Signature forgery}: a data op is signed legitimately, then its payload bytes are mutated after the signature is computed. \texttt{Op::verify} recomputes the canonical-CBOR hash over the header, observes that it no longer matches the recorded \texttt{op\_id}, and the op is dropped at ingest before reaching the deny-wins gate. The bench injects one such op alongside one in-window legitimate write and confirms that exactly the legitimate write applies. (b)~\textbf{Fabricated HLC}: the attacker signs a syntactically valid op whose HLC is fabricated to lie strictly after the credential's \texttt{exp\_ms}. The op ingests cleanly (signature is valid) but the gate denies it because the epoch's HLC window has elapsed. (c)~\textbf{Duplicate \texttt{op\_id}}: the same op is inserted twice into the merged DAG. \texttt{Dag::insert} short-circuits on a known \texttt{op\_id}, so the underlying op materialises exactly once. \subsection{Limitations and Threats to Validity}\label{subsec:limitations}

Disconnected nodes cannot learn of revocations until they sync~\cite{10.1007/978-3-030-01950-1_48}. The vulnerability window equals partition duration; operations written by a revoked user are invalidated on reconnection, but not before. The append-only log grows monotonically, and the full log must be retained for audit. Log compaction with verifiable snapshots could bound storage but remains future work. Long-offline nodes face a proportionally large replay on reconnection; checkpoints help but do not eliminate this. Per-tag encryption protects payloads, yet operation structure, timestamps, and producer keys remain visible; full metadata privacy would require anonymous credentials, Attribute-Based Encryption (ABE), or zero-knowledge proofs~\cite{8888155,zhang2020attribute}. Revocation also cannot undo physical-world actions already taken (e.g., an actuator that has fired). Applications needing staged commit must handle this themselves.

Regarding validity, E6 covers six log sizes with five trials each, but E8 and E9 use fewer sizes and a single workload. The workstation numbers come from one Linux x86\_64 machine; E15 covers edge-device replay on Raspberry Pi 4-class hardware; E16 replaces the E12 in-process transport with a real libp2p stack under network emulation; E17 adds a domain-shaped IoT-fleet workload. The workloads remain synthetic (E17 adds a domain-shaped generator, but validation on production traces is still future work), and mechanized verification of the theorems (Section~\ref{subsec:main-theorems}) is not in scope.

\section{Discussion}
\label{sec:discussion}

\paragraph*{\textit{Partition-duration trade-off}}
\equack{}'s deny-wins gate repairs authorization state after partition heal but cannot prevent unauthorized effects during the partition itself. The vulnerability window equals partition duration ($W(r) \leq \Delta_{\text{partition}} + \Delta_{\text{propagation}}$), a revoked user's writes are retroactively invalidated when logs merge, yet physical-world consequences (actuator commands, parameter changes) persist. The guarantee is that the final digital state $s^\star$ is always consistent with the revocation, the physical world cannot be rewound. For short disconnections (minutes to hours), operators can defer safety-critical actuations until sync confirms authorization; for day-long partitions, application-layer compensating controls (staged commit, human-in-the-loop) are needed.

\paragraph*{\textit{Log compaction sketch}}
The append-only log is the source of truth for replay; without compaction it grows monotonically (E8: $\approx$430~B/op CBOR, $\approx$800~B/op on disk after RocksDB amplification, so a 500K-op deployment occupies $\approx$555~MB). The \texttt{store::checkpoint} module already produces verifiable state snapshots used by E6's incremental replay path; extending it to a compaction primitive requires only tying the snapshot to a DAG \emph{watermark}: a frontier of $\textsf{op\_id}$s such that every event causally before the frontier has been incorporated into the snapshot's state digest. Once a watermark is anchored (signed by the same stakeholders that sign the genesis, and pinned in the audit chain), events strictly before it can be pruned from local storage; new replicas bootstrap by fetching the snapshot, the watermark anchor, and the suffix DAG. Convergence (Theorem~2) is preserved because the snapshot replaces a deterministic prefix replay with its byte-identical result, and audit tamper-evidence (Theorem~4) is preserved because the audit-chain head pins the watermark. As an order-of-magnitude estimate, a 500K-op deployment compacted to a $\sim$50K-op suffix plus a $\sim$200~B snapshot would shrink from $\approx$555~MB to $\approx$60~MB (an $\approx$9$\times$ reduction), with the per-replica replay window proportionally reduced. Compaction is not implemented in the prototype; the open design issues are watermark-anchor governance (which stakeholders sign, how often) and how to handle long-disconnected nodes whose own un-merged events predate a published watermark.

\paragraph*{\textit{Hold-and-confirm pattern for safety-critical actuation}}
For deployments where physical-world actions are irreversible (e.g., actuator commands, safety-parameter changes applied to running equipment), we recommend a two-phase pattern: (1)~the controller records the requested change in the event log and applies it to a \emph{pending} state visible to the local operator, but (2)~defers commitment to the \emph{operational} state until the next sync cycle confirms that the underlying credential remains valid in the converged replay and no intervening \textsc{REVOKE} has closed the epoch. This hold-and-confirm pattern is not part of \equack{}'s model; it is an application-layer policy that uses \equack{}'s replay verdicts as input. The trade-off is availability: pending changes are visible but not yet authoritative, which may be acceptable for scheduled maintenance windows but not for emergency operations. A variant for short disconnections lowers the latency floor by pairing hold-and-confirm with a local ``freshness budget'' (a maximum tolerable age of the status evidence), beyond which the controller degrades to a safe fallback mode instead of acting optimistically.

\paragraph*{\textit{Scalability}}
Memory grows at $\approx$1.7~KB/op (Table~\ref{tab:e9-memory}), so a 256\,MB edge device holds roughly 150K operations, enough for weeks of maintenance logs, but not high-frequency telemetry. Throughput drops from 45K~ops/s on linear chains to 7K~ops/s with eight concurrent writers (Table~\ref{tab:e7-throughput}); at $\approx$430~bytes/op in CBOR, a million-operation log occupies $\sim$430\,MB before write amplification. Without log compaction (deferred to future work), long-lived deployments will need archival. These constraints position \equack{} for authorization-rate workloads on devices with modest storage rather than high-throughput data-plane enforcement.

\paragraph*{\textit{Deny-wins conservatism}}
The deny-wins semantics are deliberately conservative; concurrent GRANT and REVOKE resolves in favor of revocation, appropriate where over-granting is riskier than over-denying. An erroneous revocation retroactively voids subsequent work; recovery requires a new GRANT to open a fresh epoch (Section~\ref{subsec:formal-model}), which may not be timely during disconnection. Making the combining algorithm configurable per scope is a natural extension, though it would complicate the convergence argument.

\paragraph*{\textit{Clock skew}}
\equack{} relies on the physical component of the HLC for credential validity checks. In deployments without trusted time sources (e.g., GPS, Network Time Protocol (NTP)), clock skew can cause a credential to appear valid on one replica and expired on another for events near validity boundaries. The formal property (Theorem~1) is unaffected: determinism depends on the HLC values recorded in the log, not on their accuracy relative to real time. However, the \emph{intended} authorization semantics---that a credential should not be usable outside its validity interval---degrade proportionally to clock skew. Deployments should either provision trusted time sources or set validity margins exceeding the expected skew bound $\epsilon$ (Section~\ref{subsec:system-model}, item~7). Operators facing drift-prone edge devices can also reduce boundary ambiguity by issuing credentials with coarse-grained (e.g., hour- or day-level) validity windows relative to the expected skew.

\paragraph*{\textit{In-band trust}}
Carrying issuer keys and status-list updates as replicated log events eliminates online PKI lookups, enabling offline VC verification. The cost is that TrustView grows with each key registration and status-list chunk; deployments with frequent key rotations across many issuers would see meaningful log growth from trust material alone. The pinned genesis checkpoint anchors the trust chain; its governance (who signs, how keys are distributed, how organizations join or leave) is an open design problem.

\paragraph*{\textit{Genesis attack surface}}
The genesis checkpoint is the trust root of every \equack{} deployment, so its compromise invalidates every downstream guarantee. If an attacker substitutes a genesis carrying their own issuer keys before any victim replica pins it, that replica's TrustView accepts attacker-issued credentials as legitimate; Theorems~1--3 still hold over the resulting (corrupted) DAG, but the deny-wins gate is now defending the wrong perimeter. The relevant attack surface is therefore the out-of-band distribution channel for the genesis digest, not the replay machinery itself. We recommend three layered mitigations: (1)~\emph{multi-party cross-signing}, in which each founding stakeholder independently signs the genesis and replicas accept it only when at least $k$-of-$n$ signatures verify, raising the bar from one compromised key to $k$; (2)~\emph{public anchoring}, in which the genesis digest is pinned in a tamper-evident transparency log or a public-blockchain anchor, used not for consensus but for non-equivocation, so that any divergence between the digest a replica saw and the digest the rest of the world saw is publicly detectable; and (3)~\emph{hardware-backed key custody} for the genesis-signing keys themselves (HSMs or smartcards with audited firmware), reducing the chance of silent key extraction. The prototype's single-admin genesis is a development convenience and should be read as the minimum-viable bootstrap, not a production recommendation. \emph{Split-brain genesis}, in which two disjoint groups of stakeholders each produce a self-consistent but incompatible genesis, is the same problem in another form: the result is two non-interoperable trust domains whose existence cannot be detected from inside either domain. Detection requires out-of-band comparison of pinned digests (either via the public anchor in mitigation~(2) or by direct human governance), and resolution requires human action: the protocol cannot automatically reconcile incompatible trust roots, because doing so would require trusting one side over the other --- exactly the decision the genesis was supposed to settle in the first place.

\paragraph*{\textit{Online-mode authorization latency}}
\equack{} is engineered for the offline case, but the same gate runs online. Isolating a single $\mathsf{Gate}$ decision (policy check + epoch lookup, no signature verification, no DAG insertion, no CRDT merge) in the \texttt{bench\_gate} microbenchmark over $10{,}000$ iterations after a $1{,}024$-iteration warm-up, we measure $\approx$96~ns mean per call, with p50/p95/p99 of 95/102/103~ns and a worst-case observed value of 5.9~$\mu$s on the bench host (one IssuerKey + Credential + Grant in the index, single granted subject, scope $\{\text{hv},\text{mech}\}$). At $\sim$$10^{-7}$~s per decision, the gate adds roughly an order of magnitude over a primary-key RBAC table lookup but lands in the same regime as Cedar's reported evaluation latency on policies of comparable size. Deployments with reliable connectivity, a single trust authority, and no requirement for retroactive revocation safety should still prefer a centralised policy engine; \equack{}'s value lies in workloads where the offline guarantees (R1, R3, R5, Section~\ref{subsec:conceptual-requirements}) are needed.

\paragraph*{\textit{Comparison scope}}
Tables~\ref{tab:gap-summary} and~\ref{tab:mech-compare} evaluate \equack{} against paradigm categories rather than specific systems, because we are not aware of an existing system that combines CRDT replication with in-band VCs and a deny-wins gate. The key distinction from application-level access control is enforcement timing; application-level checks execute at ingest and cannot retroactively invalidate operations accepted under stale evidence; \equack{}'s replay-time gate can (R3). The closest system is p2panda, whose retroactive replay of application data is optional and application-implemented, whereas \equack{}'s deny-wins gate is mandatory during deterministic replay. A direct empirical comparison is infeasible as p2panda provides no peer-reviewed formal model or published benchmarks.

\paragraph*{\textit{Scoped replication}}
The prototype assumes \emph{full replication}: every replica eventually receives every event in the DAG. For deployments where this is impractical (e.g., 50 controllers $\times$ 100K ops/each, where no single replica needs the union), \emph{scoped replication} is a natural extension. The DAG structure already supports it: a replica can maintain a subgraph rooted at the resource tags it manages and gossip only events whose payload tags match a configured subscription filter. Scoped replication does not affect Theorem~2 (Convergence) for the subscribed subgraph, because the proof is over the subset of events a given replica actually ingests; what it requires is careful handling of cross-scope \textsc{REVOKE} events (a credential revocation that closes epochs spanning more than one tag must reach every replica subscribed to any of those tags) and of TRUST events (issuer keys and status-list chunks have to flow to every replica that might verify a credential touching the affected scope). Designing the subscription topology so these cross-cutting events propagate without re-introducing full replication is left for future work.

\paragraph*{\textit{Cedar integration roadmap}}
The current policy engine is a $\sim$1{,}000-LOC purpose-built evaluator (\texttt{core::policy}) covering the Cedar subset enumerated alongside Listing~\ref{lst:cedar-fleet}. It exists because the prototype predates the stabilisation of the upstream Cedar Rust SDK (\texttt{cedar-policy} crate). Substituting the upstream SDK as a drop-in replacement for the evaluator is feasible: \equack{}'s gate already calls the policy engine through a single internal interface (subject + action + resource tags + epoch index + position $\rightarrow$ allow/deny), and the upstream SDK provides exactly that API plus full type checking, schema validation, partial evaluation, and the verified properties cited by the Cedar team. The migration is on the roadmap but is not prioritised over the mechanised verification work for the replay--gate composition (Theorems~1--3); the latter is a paradigm-level concern that the upstream SDK cannot answer for us, whereas the former is a purely engineering task that can be done at any time without affecting the formal claims.

\paragraph*{\textit{Prototype scope and Cedar properties}}
Cedar's verified properties (decidability, totality, soundness) hold for the converged state $s^\star$ but not for transient local decisions during partitions, an inherent limitation of any eventually consistent system embedding a single-context policy engine. The prototype implements the full conceptual model with simplifications; VC validation omits JSON-LD and selective disclosure; key management uses X25519 without a full ceremony protocol; Cedar's full type system is not implemented; and libp2p gossip uses basic flooding. The evaluation uses synthetic workloads on a single Linux x86\_64 machine; E12 simulates multi-node sync in-process; cross-platform determinism is confirmed between x86\_64 and aarch64 (E15); and the confidentiality layer is experimental. The formal claims (Theorems~1--5) are argued but not mechanically verified; mechanized verification of the replay-gate composition is a priority for future work.

\section{Conclusion}
\label{sec:conclusion}

Offline multi-stakeholder systems need authorization decisions that remain available during disconnection and converge deterministically after partition healing. The ECAC paradigm~\cite{jacob2024eventuallyac} establishes that such convergence is formally achievable, but leaves open how to realize it with concrete credentials, fine-grained policies, and in-band trust. \equack{} closes this gap by recording data operations, credential changes, and trust evidence in a single signed causal event log and reconstructing authorized state through deterministic deny-wins replay.

The prototype evaluation supports this within a single-machine, synthetic-workload setup. One hundred valid orderings of a 10\,000-operation log all converged to the same state digest; a revocation workload suppressed every post-revocation write after reconciliation; multi-issuer conflicts resolved deterministically; and audit tampering was detected. Replay throughput reached 45\,000~operations/s on linear chains, and checkpoint restore cut replay cost by 8--11$\times$.

A credential revocation issued by any authorized issuer eventually takes effect at every replica, regardless of partition topology or message ordering. The in-band trust model eliminates the dependency on always-online PKI infrastructure, and the tamper-evident audit trail allows independent parties to verify that authorization decisions were computed correctly from the same evidence. Together, these properties enable offline authorization in settings, industrial IoT, federated data sharing, disaster response, where, to the best of our knowledge, no prior system surveyed in Sections~\ref{subsec:gap-analysis}--\ref{subsec:mechanism-comparison} simultaneously provides deterministic convergence, retroactive revocation safety, and fine-grained policy expressiveness.

The workloads are synthetic, the multi-node topology is simulated in-process, and the confidentiality layer is experimental. The model trades real-time revocation for partition availability, it guarantees the converged \emph{digital} state, not reversal of physical-world actions. Multi-machine evaluation under realistic network partitions and mechanized verification of the replay-gate composition are the priority next steps.
